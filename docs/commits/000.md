# 000 — chore(project): initialize odds-feed-service layout

## 개요

odds-feed-service의 골격을 세운 첫 커밋. 비즈니스 로직은 0이고, **빌드 정의 +
profile + 관측성 wiring + 빈 패키지 레이아웃**만 깐다. 이 repo는 Phase 2 leaf —
`shared-protocol`만 의존하고, betting-service가 Redis로 odds를 읽기 때문에
betting보다 먼저 와야 한다.

핵심 결정 하나: **이 repo는 라이브러리가 아니라 application이다.** 그래서
`shared-protocol`처럼 BOM만 import하는 게 아니라 `spring-boot-starter-parent`를
**상속(parent)** 한다. application은 실행 가능한 fat jar와 `spring-boot:run`이
필요하고, 그걸 `spring-boot-maven-plugin`이 해주는데 그 plugin 관리가 parent에
들어 있다.

> **C/C++ 개발자를 위한 메모.** Maven의 `<parent>`는 C++엔 직접 대응물이 없다.
> CMake에서 공통 toolchain·플래그를 상위 파일에서 상속하는 것과 비슷하다고 보면
> 된다. `spring-boot-starter-parent`는 수백 개 라이브러리의 **버전을 한 번에
> 고정**(dependency management)하고, 빌드 plugin 기본 설정을 내려준다. 그래서
> 우리 `pom.xml`의 `<dependency>`들은 대부분 `<version>`을 안 적는다 — parent가
> 정해주니까.

## 작업 순서

1. `git init -b main`, 디렉터리 골격 생성.
2. `shared-protocol`에서 Maven wrapper(`mvnw`)·checkstyle 설정을 복사.
3. `.gitignore` 작성 — **`CLAUDE.md` 라인 필수**.
4. `pom.xml` — parent, 의존성, 빌드 plugin.
5. profile 3종 (`application.yml` / `-mock` / `-real`) + `logback-spring.xml`.
6. `OddsFeedApplication` + 빈 smoke test.
7. `docs/{commits,notes,reflection}/.gitkeep`, `load-test/.gitkeep`.
8. `./mvnw verify`로 빌드·포맷·린트 통과 확인.

## 작업 내역

### 빌드 정의 (pom.xml)

application이므로 parent를 상속한다 (개요의 핵심 결정):

```xml
<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>3.2.11</version>   <!-- ADR-0015: Java 17 + Spring Boot 3.2.x -->
  <relativePath/>
</parent>
```

런타임 의존성은 이 서비스의 **표면적**에 맞춰 골랐다 — 각 줄이 곧 "이 서비스가
무엇을 하는가"다:

```xml
<dependencies>
  <!-- 공유 프로토콜: Avro 이벤트 POJO + EventId/MarketId/Odds 값 객체 + 도메인 enum. -->
  <dependency>
    <groupId>com.sportsbook</groupId>
    <artifactId>shared-protocol</artifactId>
    <version>${shared-protocol.version}</version>
  </dependency>

  <!-- REST read API + trader admin: servlet 스택으로 HTTP 서빙. -->
  <dependency>...spring-boot-starter-web</dependency>
  <!-- WebFlux는 real provider의 WebClient 전용. HTTP를 서빙하려는 게 아니다. -->
  <dependency>...spring-boot-starter-webflux</dependency>
  <!-- write-through 캐시 (Lettuce, non-blocking). -->
  <dependency>...spring-boot-starter-data-redis</dependency>
  <!-- Avro 이벤트 publish. partition key = eventId (ADR-0006). -->
  <dependency>...spring-kafka</dependency>
  <!-- liveness/readiness probe + prometheus scrape (ADR-0007). -->
  <dependency>...spring-boot-starter-actuator</dependency>
  <!-- 관측성 (ADR-0007): JSON 로그 / metrics / OTel tracing. -->
  <dependency>...logstash-logback-encoder (${logstash-encoder.version})</dependency>
  <dependency>...micrometer-registry-prometheus</dependency>
  <dependency>...micrometer-tracing-bridge-otel</dependency>
  <dependency>...opentelemetry-exporter-otlp</dependency>
  <!-- boilerplate 축소, compile-only. -->
  <dependency>...lombok (optional)</dependency>
  <!-- @Valid 요청 바디 검증. -->
  <dependency>...spring-boot-starter-validation</dependency>

  <!-- test: JUnit5+AssertJ+Mockito / embedded kafka / reactor-test /
       testcontainers(junit-jupiter, kafka, spring-boot-testcontainers). -->
  <dependency>...spring-boot-starter-test (test)</dependency>
  <dependency>...spring-kafka-test (test)</dependency>
  <dependency>...reactor-test (test)</dependency>
  <dependency>...testcontainers junit-jupiter / kafka / spring-boot-testcontainers (test)</dependency>
</dependencies>
```

빌드 plugin 3종 — 셋 다 `verify` phase 게이트라 포맷·린트 위반이 빌드를 깬다:

```xml
<build><plugins>
  <!-- 1) 실행 가능 fat jar + spring-boot:run. lombok은 최종 jar에서 제외. -->
  <plugin>spring-boot-maven-plugin  (excludes: lombok)</plugin>
  <!-- 2) google-java-format. shared-protocol과 같은 설정으로 시스템 전체 일치.
         check at verify; `mvn spotless:apply`로 자동 포맷. -->
  <plugin>spotless-maven-plugin  (googleJavaFormat GOOGLE, removeUnusedImports, check)</plugin>
  <!-- 3) 의미 규칙(magic number, parameter 수, unused import).
         config/checkstyle/checkstyle.xml 을 shared-protocol과 공유. -->
  <plugin>maven-checkstyle-plugin  (verify phase, failOnViolation=true, src/main/java only)</plugin>
</plugins></build>
```

`web`은 서빙, `webflux`는 WebClient 전용으로 **역할을 가른다**. Spotless·Checkstyle은
경고가 아니라 **빌드 게이트** — CI에서 포맷 드리프트가 누적되지 않게 처음부터 막는다.

### profile 분리 (ADR-0010)

`application.yml` — 모든 profile 공통 (Redis/Kafka 접속, 토픽명, threshold, TTL,
Actuator, OTel):

```yaml
spring:
  profiles:
    default: mock                 # 키 없는 fresh checkout도 바로 부팅
  data.redis:
    host: ${REDIS_HOST:localhost}
    timeout: 2s
    lettuce.pool: { enabled: true, max-active: 16, max-idle: 8, min-idle: 2 }
  kafka.producer:
    acks: all
    properties: { enable.idempotence: true, compression.type: lz4 }
    key-serializer: StringSerializer
    value-serializer: ByteArraySerializer   # Avro는 KafkaConfig @Bean으로 교체
server: { port: ${SERVER_PORT:8085}, shutdown: graceful }
management:
  endpoints.web.exposure.include: health,info,prometheus
  endpoint.health.probes.enabled: true      # liveness/readiness
  tracing.sampling.probability: ${OTEL_SAMPLING_PROBABILITY:1.0}
  otlp.tracing.endpoint: ${OTEL_EXPORTER_OTLP_ENDPOINT:http://localhost:4318/v1/traces}
oddsfeed:
  publish.odds-change-threshold: 0.01        # sub-1% 노이즈 필터 (CLAUDE.md)
  kafka.topics: { odds-changed: odds.changed, market-status-changed: market.status.changed,
                  event-lifecycle: event.lifecycle, match-result: match.result }
  cache.ttl: 24h
```

value-serializer를 일부러 `ByteArraySerializer`로 둬서, KafkaConfig가 Avro
serializer를 주입하기 전이라면 misconfiguration이 **조용히 String을 보내는 대신
시끄럽게 깨지게** 했다.

`application-mock.yml` (기본 profile) — 시뮬레이터 파라미터:

```yaml
oddsfeed:
  provider.mode: mock
  mock:
    minutes-per-second: 1                    # 1초=1mock분. load test는 60
    scenarios: { auto-rotate: true, rotation-interval-seconds: 60 }
    base-home-win-probability: 0.45          # 합 1.0, margin 0 (V1)
    base-draw-probability: 0.25
    base-away-win-probability: 0.30
```

`application-real.yml` — TheOddsApiProvider 활성:

```yaml
oddsfeed:
  provider.mode: real
  real:
    api-key: ${THE_ODDS_API_KEY}
    base-url: https://api.the-odds-api.com/v4
    sport-keys: [soccer_epl, basketball_nba] # quota 보호 위해 좁게
    rate-limit.max-requests-per-minute: 5    # 500 req/month 무료 티어 보호
    monthly-quota: 500
```

`logback-spring.xml`은 두 출력 모드를 Spring profile로 가른다 — `json` profile을
붙이면 `LogstashEncoder`(MDC traceId/spanId + service 태그), 안 붙이면 로컬
가독용 콘솔 포맷:

```xml
<springProfile name="!json">
  <include resource=".../console-appender.xml"/>   <!-- 로컬 grep-friendly -->
</springProfile>
<springProfile name="json">
  <appender name="JSON" class="...ConsoleAppender">
    <encoder class="net.logstash.logback.encoder.LogstashEncoder">
      <includeMdcKeyName>traceId</includeMdcKeyName>
      <includeMdcKeyName>spanId</includeMdcKeyName>
      <customFields>{"service":"${appName}"}</customFields>
    </encoder>
  </appender>
</springProfile>
```

### 저장소 규약 + entry point

`.gitignore`에 **`CLAUDE.md`** 를 박았다. AI 세션 운영 컨텍스트 파일이지
프로젝트 산출물이 아니므로 git에 절대 안 올라간다 (sportsbook/CLAUDE.md "개발
단계 vs Retrospective 단계"):

```gitignore
# AI session context — operational file, not a project artifact
CLAUDE.md
```

entry point와 smoke test. smoke test는 전체 컨텍스트를 띄우지 않고 **reflection으로
`main` 존재만** 확인해 가볍게 둔다:

```java
@SpringBootApplication
@SuppressWarnings("checkstyle:HideUtilityClassConstructor")
public class OddsFeedApplication {
  public static void main(String[] args) {
    SpringApplication.run(OddsFeedApplication.class, args);
  }
}
```

```java
class OddsFeedApplicationTests {
  @Test
  void mainMethodIsInvokable() {
    assertThatCode(() -> OddsFeedApplication.class.getDeclaredMethod("main", String[].class))
        .doesNotThrowAnyException();
  }
}
```

`docs/` 하위와 `load-test/`는 `.gitkeep`만 — 모든 회고 산출물은 retrospective
단계에서 일괄 작성.

## 결과

`./mvnw verify` BUILD SUCCESS, smoke test 1개 통과, Spotless + Checkstyle 0
violation. 커밋 1개로 "빌드되고 실행 가능한 빈 Spring Boot app" 확보.

처음 `verify`는 두 번 fail했다 — (1) smoke test의 Javadoc 주석이 google-java-format
줄바꿈 규칙과 어긋나 Spotless가 막음, (2) `OddsFeedApplication`을 Checkstyle이
utility class로 오인(`HideUtilityClassConstructor`). 전자는 `spotless:apply`로,
후자는 클래스에 `@SuppressWarnings("checkstyle:HideUtilityClassConstructor")`로
해결. 이 두 마찰은 이후 모든 커밋에서 반복돼 "코드 → `spotless:apply` →
`verify`" 루틴이 됐다.
