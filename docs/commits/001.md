# 001 — feat(provider): OddsProvider abstraction and DTO types

## 개요

ADR-0010이 요구하는 **seam(이음매)** 을 만든다. `OddsProvider` 인터페이스 하나
뒤에 mock(기본)과 real(옵션) 두 adapter가 선다. 이 커밋은 **타입만** 추가하고
구현은 다음 두 커밋에서 채운다. 외부 의존을 인터페이스로 격리해 테스트
친화적으로 만드는, hexagonal/clean architecture의 외부 어댑터 패턴 그대로다.

## 작업 순서

1. `Sport` enum (FOOTBALL, BASKETBALL) — 로컬 유지.
2. `EventSummary` 레코드 — 이벤트 스냅샷.
3. `ProviderEvent` sealed interface — 변경 이벤트 3종.
4. `MatchOutcome` 레코드 — 경기 결과.
5. `OddsProvider` 인터페이스 — 3메서드.
6. 단위 테스트 — sealed-permits 계약, null guard, 방어적 복사.

## 작업 내역

### OddsProvider 인터페이스 — ADR-0010 스케치에서 의도적으로 바꾼 것

```java
public interface OddsProvider {
  List<EventSummary> listEvents(Sport sport);
  Flux<ProviderEvent> streamEvents(EventId eventId);
  Optional<MatchOutcome> getMatchResult(EventId eventId);
}
```

- **`listEvents`는 `Stream`이 아니라 `List`를 반환.** orchestrator가 한 사이클에
  스냅샷을 여러 번 훑는다(로그 + Redis diff + id별 subscribe). `Stream`은
  일회용(single-use)이라 다시 못 쓴다. real adapter의 GET 응답이 배열인 것과도
  맞다.

  > **C/C++ 메모.** Java `Stream`은 C++ range/view처럼 lazy하지만 **재사용
  > 불가**(한 번 terminal operation을 쓰면 닫힘). 컬렉션을 여러 번 순회할 거면
  > `List`로 materialize하는 게 정석.

- **`streamEvents`는 `Flux<ProviderEvent>`.** 세 종류 변경(odds/market
  status/lifecycle)을 **하나의 순서 있는 스트림**으로 흘린다. 셋으로 쪼개면
  consumer가 발생 순서를 재조립해야 하고, 상류 구독 수가 3배가 된다.

  > **C/C++ 메모.** `Flux<T>`(Project Reactor)는 0..N개 비동기 아이템 스트림 —
  > "미래에 여러 번 값이 올 수 있는 채널". `Mono<T>`는 0..1개. RxCpp의
  > `observable`에 가깝다.

- **`getMatchResult`의 반환은 `MatchOutcome`** (Avro 생성 클래스 `MatchResult`와
  이름 충돌 회피 — 로컬 레코드는 publisher 경계에서 Avro POJO로 변환).

### ProviderEvent — sealed interface (Java 17)

```java
public sealed interface ProviderEvent
    permits ProviderEvent.OddsUpdated,
            ProviderEvent.MarketStatusUpdated,
            ProviderEvent.LifecycleUpdated { ... }
```

`permits`로 구현 가능한 타입을 **컴파일 타임에 봉인**한다. publisher가 이
sealed 타입을 보고 세 Avro 토픽으로 dispatch한다.

> **C/C++ 메모.** sealed interface는 닫힌 타입 집합 — Rust `enum`이나 C++
> `std::variant`의 타입-안전 버전에 가깝다. `record`는 불변 데이터 캐리어
> (C++ `struct` + 자동 생성 ctor/getter/equals/hashCode). 둘을 합치면 컴파일러가
> "이 셋이 전부"임을 알아 exhaustiveness를 도와준다.

### shared-protocol 타입 재사용

- `Odds`(BigDecimal + scale 정규화 equality, ADR-0003) — JSON/Avro round-trip이
  trailing zero를 깎아도 equality가 유지됨.
- `EventId` / `MarketId` / `SelectionId` — UUID에 **컴파일 타임 타입 구분**.
  `MarketId`를 `EventId` 자리에 넣으면 런타임이 아니라 컴파일이 막는다.
- `MarketStatus` / `EventLifecycleStatus` / `MatchFinalStatus` — Avro 생성 enum
  재사용으로 wire 어휘와 안 어긋나게.

레코드들은 compact constructor에서 non-null 검증하고, `MatchOutcome.detail`은
입력 맵을 `Map.copyOf`로 방어 복사한다.

## 결과

9개 테스트 통과 — sealed permits 정확히 3종, 각 레코드 null guard, 방어 복사
불변성. 인터페이스 변경 없이 다음 두 커밋이 구현만 끼울 수 있는 안정된 seam 확보.

## 요약

- `OddsProvider` 3메서드, mock/real 공유.
- `ProviderEvent` sealed로 변경 3종을 한 `Flux`에.
- `Stream` 대신 `List`, 이름 충돌 회피 `MatchOutcome`.
- `Sport`는 로컬 enum (다른 서비스는 eventId만 봄 → shared-protocol 미오염).

## 기억·설명 Level

- **L1**: `Sport` enum 값, 레코드 필드 목록.
- **L2**: shared-protocol 타입 재사용 이유, `MatchOutcome` 네이밍 충돌.
- **L3**: **`List` vs `Stream` 반환 선택**, **sealed interface + record로
  변경 이벤트 모델링**, **단일 `Flux` vs 3-스트림** trade-off. adapter 패턴이
  왜 테스트·확장에 유리한지. 면접에서 "외부 의존 격리" 단골.
