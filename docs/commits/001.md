# 001 — feat(provider): OddsProvider abstraction and DTO types

## 개요

ADR-0010이 요구하는 **seam(이음매)** 을 만든다. `OddsProvider` 인터페이스 하나
뒤에 mock(기본)과 real(옵션) 두 adapter가 선다. 이 커밋은 **타입만** 추가하고
구현은 다음 두 커밋에서 채운다. 외부 의존을 인터페이스로 격리해 테스트
친화적으로 만드는, hexagonal/clean architecture의 외부 어댑터 패턴 그대로다.

## 작업 순서

1. `Sport` enum (FOOTBALL, BASKETBALL) — 로컬 유지.
2. `EventSummary` 레코드 — 이벤트 스냅샷.
3. `ProviderEvent` sealed interface — 변경 이벤트 3종.
4. `MatchOutcome` 레코드 — 경기 결과.
5. `OddsProvider` 인터페이스 — 3메서드.
6. 단위 테스트 — sealed-permits 계약, null guard, 방어적 복사.

## 작업 내역

### OddsProvider 인터페이스 — ADR-0010 스케치에서 의도적으로 바꾼 것

```java
public interface OddsProvider {
  List<EventSummary> listEvents(Sport sport);
  Flux<ProviderEvent> streamEvents(EventId eventId);
  Optional<MatchOutcome> getMatchResult(EventId eventId);
}
```

- **`listEvents`는 `Stream`이 아니라 `List`를 반환.** orchestrator가 한 사이클에
  스냅샷을 여러 번 훑는다(로그 + Redis diff + id별 subscribe). `Stream`은
  일회용(single-use)이라 다시 못 쓴다. real adapter의 GET 응답이 배열인 것과도
  맞다.

  > **C/C++ 메모.** Java `Stream`은 C++ range/view처럼 lazy하지만 **재사용
  > 불가**(한 번 terminal operation을 쓰면 닫힘). 컬렉션을 여러 번 순회할 거면
  > `List`로 materialize하는 게 정석.

- **`streamEvents`는 `Flux<ProviderEvent>`.** 세 종류 변경(odds/market
  status/lifecycle)을 **하나의 순서 있는 스트림**으로 흘린다. 셋으로 쪼개면
  consumer가 발생 순서를 재조립해야 하고, 상류 구독 수가 3배가 된다. 추적하지
  않는 이벤트면 `Flux.empty()`. consumer는 hot stream으로 가정(늦게 구독하면
  누락 가능).

  > **C/C++ 메모.** `Flux<T>`(Project Reactor)는 0..N개 비동기 아이템 스트림 —
  > "미래에 여러 번 값이 올 수 있는 채널". `Mono<T>`는 0..1개. RxCpp의
  > `observable`에 가깝다.

- **`getMatchResult`의 반환은 `MatchOutcome`** (Avro 생성 클래스 `MatchResult`와
  이름 충돌 회피 — 로컬 레코드는 publisher 경계에서 Avro POJO로 변환).
  `LifecycleUpdated`가 `FINISHED`를 나르면 orchestrator가 폴링한다.

### ProviderEvent — sealed interface (Java 17)

`permits`로 구현 가능한 타입을 **컴파일 타임에 봉인**한다. 변경 3종을 record로
중첩하고, 공통 접근자 두 개(`eventId`, `occurredAt`)를 인터페이스에 둔다:

```java
public sealed interface ProviderEvent
    permits ProviderEvent.OddsUpdated,
        ProviderEvent.MarketStatusUpdated,
        ProviderEvent.LifecycleUpdated {

  EventId eventId();
  Instant occurredAt();

  /** 한 selection 가격 이동. publish 시 1% 노이즈 임계값 적용. */
  record OddsUpdated(
      EventId eventId, MarketId marketId, SelectionId selectionId,
      Odds previousOdds, Odds newOdds, Instant occurredAt)
      implements ProviderEvent {
    public OddsUpdated {
      Objects.requireNonNull(eventId, "eventId");
      Objects.requireNonNull(marketId, "marketId");
      Objects.requireNonNull(selectionId, "selectionId");
      Objects.requireNonNull(previousOdds, "previousOdds");
      Objects.requireNonNull(newOdds, "newOdds");
      Objects.requireNonNull(occurredAt, "occurredAt");
    }
  }

  /** 마켓 open/suspend/close. reason은 선택적(null 허용). */
  record MarketStatusUpdated(
      EventId eventId, MarketId marketId,
      MarketStatus previousStatus, MarketStatus newStatus,
      String reason, Instant occurredAt)
      implements ProviderEvent {
    public MarketStatusUpdated {
      Objects.requireNonNull(eventId, "eventId");
      Objects.requireNonNull(marketId, "marketId");
      Objects.requireNonNull(previousStatus, "previousStatus");
      Objects.requireNonNull(newStatus, "newStatus");
      Objects.requireNonNull(occurredAt, "occurredAt");   // reason은 검증 안 함
    }
  }

  /** 이벤트 자체의 lifecycle 전이(kickoff, 종료, 취소). */
  record LifecycleUpdated(
      EventId eventId, EventLifecycleStatus status,
      Instant scheduledStartAt, Instant occurredAt)
      implements ProviderEvent {
    public LifecycleUpdated {
      Objects.requireNonNull(eventId, "eventId");
      Objects.requireNonNull(status, "status");
      Objects.requireNonNull(scheduledStartAt, "scheduledStartAt");
      Objects.requireNonNull(occurredAt, "occurredAt");
    }
  }
}
```

`reason`만 의도적으로 검증에서 뺀다(open으로 돌릴 땐 사유가 없을 수 있음).
publisher가 이 sealed 타입을 instanceof로 분기해 세 Avro 토픽으로 dispatch한다
(wire는 토픽별로 쪼개져 있지만, in-process는 순서 보존 위해 한 스트림).

> **C/C++ 메모.** sealed interface는 닫힌 타입 집합 — Rust `enum`이나 C++
> `std::variant`의 타입-안전 버전에 가깝다. `record`는 불변 데이터 캐리어
> (C++ `struct` + 자동 생성 ctor/getter/equals/hashCode). 둘을 합치면 컴파일러가
> "이 셋이 전부"임을 알아 exhaustiveness를 도와준다.

### DTO 레코드 + shared-protocol 타입 재사용

`EventSummary`(스냅샷)와 `MatchOutcome`(결과)은 compact constructor에서 non-null을
검증하고, `MatchOutcome.detail`은 입력 맵을 `Map.copyOf`로 **방어 복사**한다:

```java
public record EventSummary(
    EventId eventId, Sport sport, String competition,
    String homeTeam, String awayTeam,
    Instant scheduledStartAt, EventLifecycleStatus status) {
  public EventSummary {
    Objects.requireNonNull(eventId, "eventId");
    Objects.requireNonNull(sport, "sport");
    Objects.requireNonNull(competition, "competition");
    Objects.requireNonNull(homeTeam, "homeTeam");
    Objects.requireNonNull(awayTeam, "awayTeam");
    Objects.requireNonNull(scheduledStartAt, "scheduledStartAt");
    Objects.requireNonNull(status, "status");
  }
}

public record MatchOutcome(
    EventId eventId, String score, MatchFinalStatus finalStatus,
    Map<String, String> detail, Instant settledAt) {
  public MatchOutcome {
    Objects.requireNonNull(eventId, "eventId");
    Objects.requireNonNull(score, "score");
    Objects.requireNonNull(finalStatus, "finalStatus");
    Objects.requireNonNull(detail, "detail");
    Objects.requireNonNull(settledAt, "settledAt");
    detail = Map.copyOf(detail);   // 불변·방어 복사 — 호출자가 나중에 못 바꿈
  }
}
```

- `MatchOutcome.detail`은 **schema-less `Map`** — 새 종목(yellow card, 점유율,
  테니스 세트 스코어)이 Avro 진화를 안 건드리게 일부러 자유 형식.
- `EventSummary.competition`은 자유 표시 라벨("Premier League") — V1은 리그/대회를
  1급 엔티티로 모델링 안 함(하류가 그걸로 dispatch 안 하니까).

재사용한 shared-protocol 타입:

- `Odds`(BigDecimal + scale 정규화 equality, ADR-0003) — JSON/Avro round-trip이
  trailing zero를 깎아도 equality가 유지됨.
- `EventId` / `MarketId` / `SelectionId` — UUID에 **컴파일 타임 타입 구분**.
  `MarketId`를 `EventId` 자리에 넣으면 런타임이 아니라 컴파일이 막는다.
- `MarketStatus` / `EventLifecycleStatus` / `MatchFinalStatus` — Avro 생성 enum
  재사용으로 wire 어휘와 안 어긋나게.

`Sport`만 로컬 enum으로 둔다 — 다른 서비스는 wire에서 `eventId`로만 식별하지
sport로 분기하지 않으므로 shared-protocol을 오염시키지 않는다. 두 번째 서비스가
sport로 구별해야 하는 순간 옮긴다:

```java
public enum Sport { FOOTBALL, BASKETBALL }
```

## 결과

9개 테스트 통과 — sealed permits 정확히 3종, 각 레코드 null guard, 방어 복사
불변성. 인터페이스 변경 없이 다음 두 커밋이 구현만 끼울 수 있는 안정된 seam 확보.

## 요약

- `OddsProvider` 3메서드, mock/real 공유.
- `ProviderEvent` sealed로 변경 3종을 한 `Flux`에.
- `Stream` 대신 `List`, 이름 충돌 회피 `MatchOutcome`.
- `Sport`는 로컬 enum (다른 서비스는 eventId만 봄 → shared-protocol 미오염).

- **L2**: shared-protocol 타입 재사용 이유, `MatchOutcome` 네이밍 충돌.
- **L3**: **`List` vs `Stream` 반환 선택**, **sealed interface + record로
  변경 이벤트 모델링**, **단일 `Flux` vs 3-스트림** trade-off. adapter 패턴이
  왜 테스트·확장에 유리한지. 면접에서 "외부 의존 격리" 단골.
