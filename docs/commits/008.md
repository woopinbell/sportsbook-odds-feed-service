# 008 — feat(api): REST read endpoints and trader admin endpoints

## 개요

HTTP 클라이언트를 위한 두 표면적:

```
GET  /api/v1/events                              kickoff 순 cursor pagination
GET  /api/v1/events/{id}                         단건
GET  /api/v1/odds/{eventId}/{marketId}/{selId}   현재 odds 조회

POST /internal/v1/events/{eventId}/markets/{marketId}/suspend
POST /internal/v1/events/{eventId}/markets/{marketId}/close
POST /internal/v1/events/{eventId}/markets/{marketId}/reopen
```

`/api/v1`은 고객용(gateway 라우팅), `/internal/v1`은 admin-api용. path로 가르면
미래의 API gateway가 endpoint별 애너테이션 없이 **경로 단위로 보안 정책**을
다르게 줄 수 있다.

## 작업 순서

1. `EventCatalog` — 인메모리 이벤트 인덱스.
2. `CursorPage<T>` + `EventReadController` — cursor pagination.
3. `OddsReadController` — Redis read.
4. `MarketAdminController` — suspend/close/reopen.
5. `FeedOrchestrator`에 catalog 4번째 인자로 연결.
6. `@WebMvcTest` 컨트롤러 테스트.

## 작업 내역

### EventCatalog — 왜 Redis SCAN을 안 쓰나

이벤트 목록 응답을 Redis가 아니라 인메모리 `ConcurrentHashMap`에서 읽는다.
`event:*`에 대한 Redis `SCAN`은 O(N)이고 다른 Redis 명령을 블록한다:

```java
@Component
public class EventCatalog {

  private final Map<EventId, EventSummary> events = new ConcurrentHashMap<>();

  public void put(EventSummary summary) {
    events.put(summary.eventId(), summary);
  }
  public Optional<EventSummary> get(EventId eventId) {
    return Optional.ofNullable(events.get(eventId));
  }

  /** kickoff 오름차순 스냅샷, eventId tie-break으로 커서 안정성 확보. */
  public List<EventSummary> orderedByKickoff() {
    return events.values().stream()
        .sorted(Comparator.comparing(EventSummary::scheduledStartAt)
            .thenComparing(e -> e.eventId().value()))
        .toList();
  }
  public int size() {
    return events.size();
  }
}
```

orchestrator가 refresh·lifecycle dispatch마다 catalog에 써준다. **Redis 사본은
서비스 간 read의 canonical로 남고, EventCatalog는 철저히 로컬**이다.

### cursor pagination (ADR-0004)

응답 래퍼 + 컨트롤러. 커서는 `"{kickoffInstant}|{eventUuid}"`를 URL-safe base64로
불투명하게 감싼다:

```java
public record CursorPage<T>(List<T> items, String nextCursor) {}   // nextCursor=null → 마지막 페이지
```

```java
@RestController
@RequestMapping("/api/v1/events")
public class EventReadController {

  private static final int DEFAULT_PAGE_SIZE = 20;
  private static final int MAX_PAGE_SIZE = 100;
  private final EventCatalog catalog;

  public EventReadController(EventCatalog catalog) {
    this.catalog = catalog;
  }

  @GetMapping
  public CursorPage<EventSummary> list(
      @RequestParam(value = "cursor", required = false) String cursor,
      @RequestParam(value = "size", defaultValue = "20") int requestedSize) {
    int size = clampSize(requestedSize);
    List<EventSummary> all = catalog.orderedByKickoff();
    int startIndex = cursor == null ? 0 : indexAfter(all, decodeCursor(cursor));
    int endIndex = Math.min(startIndex + size, all.size());
    List<EventSummary> page = all.subList(startIndex, endIndex);
    String nextCursor = endIndex < all.size() ? encodeCursor(page.get(page.size() - 1)) : null;
    return new CursorPage<>(page, nextCursor);
  }

  @GetMapping("/{eventId}")
  public ResponseEntity<EventSummary> get(@PathVariable("eventId") UUID eventId) {
    return catalog.get(new EventId(eventId))
        .map(ResponseEntity::ok)
        .orElseThrow(() -> new ResponseStatusException(HttpStatus.NOT_FOUND, "Event not found"));
  }

  static int clampSize(int requestedSize) {
    if (requestedSize <= 0) return DEFAULT_PAGE_SIZE;
    return Math.min(requestedSize, MAX_PAGE_SIZE);          // [1, 100], 기본 20
  }

  static String encodeCursor(EventSummary summary) {
    String raw = summary.scheduledStartAt().toString() + "|" + summary.eventId().value();
    return Base64.getUrlEncoder().withoutPadding().encodeToString(raw.getBytes(StandardCharsets.UTF_8));
  }

  static Cursor decodeCursor(String encoded) {
    try {
      String raw = new String(Base64.getUrlDecoder().decode(encoded), StandardCharsets.UTF_8);
      int sep = raw.indexOf('|');
      if (sep < 0) throw new IllegalArgumentException("missing separator in cursor");
      return new Cursor(Instant.parse(raw.substring(0, sep)), UUID.fromString(raw.substring(sep + 1)));
    } catch (IllegalArgumentException | DateTimeParseException e) {
      throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "Invalid cursor", e);   // 400
    }
  }

  private static int indexAfter(List<EventSummary> all, Cursor cursor) {
    for (int i = 0; i < all.size(); i++) {
      EventSummary s = all.get(i);
      int kickoffCompare = s.scheduledStartAt().compareTo(cursor.kickoff());
      if (kickoffCompare > 0) return i;
      if (kickoffCompare == 0 && s.eventId().value().compareTo(cursor.eventId()) > 0) return i;
    }
    return all.size();
  }

  record Cursor(Instant kickoff, UUID eventId) {}
}
```

- 정렬은 kickoff 오름차순 + `eventId` tie-break이라 같은 Instant의 중복이 커서
  seek을 안 깨뜨린다.
- 페이지 크기 `[1, 100]` clamp, 기본 20. 잘못된 커서 → 400(파싱 에러 첨부),
  스트림 소진 → `nextCursor = null`.

> **C/C++ 메모.** cursor pagination은 offset/limit 대신 "마지막으로 본 키"를
> 들고 다음 페이지를 seek한다. 시계열 데이터에서 offset이 가진 문제(중간에 행이
> insert되면 페이지가 밀림)를 피한다. 커서를 base64로 불투명하게 만들어
> 클라이언트가 내부 구조에 의존 못 하게 한다.

### OddsReadController — URL이 3-튜플을 다 받는 이유

```java
@RestController
@RequestMapping("/api/v1/odds")
public class OddsReadController {

  private final RedisOddsCache cache;
  public OddsReadController(RedisOddsCache cache) { this.cache = cache; }

  @GetMapping("/{eventId}/{marketId}/{selectionId}")
  public OddsResponse getOdds(
      @PathVariable("eventId") UUID eventId,
      @PathVariable("marketId") UUID marketId,
      @PathVariable("selectionId") UUID selectionId) {
    Odds odds = cache.getOdds(new EventId(eventId), new MarketId(marketId), new SelectionId(selectionId))
        .orElseThrow(() -> new ResponseStatusException(HttpStatus.NOT_FOUND, "Odds not found"));
    return new OddsResponse(eventId, marketId, selectionId, odds.decimal());
  }

  public record OddsResponse(UUID eventId, UUID marketId, UUID selectionId, BigDecimal odds) {}
}
```

reverse index(marketId → eventId)를 V1에 안 둬서, Redis가 (eventId, marketId,
selectionId) 복합키로 묶여 있다. 그래서 URL이 셋을 다 받는 게 싼 길이다. 실제
호출자(betting-service)가 eventId를 못 들고 오는 게 확인될 때 reverse index를
추가하면 된다.

### MarketAdminController — 세 endpoint가 같은 일

```java
@RestController
@RequestMapping("/internal/v1/events/{eventId}/markets/{marketId}")
public class MarketAdminController {

  private final RedisOddsCache cache;
  private final OddsFeedPublisher publisher;
  private final Clock clock;
  // 생성자 생략 (cache, publisher, clock 주입)

  @PostMapping("/suspend") @ResponseStatus(HttpStatus.ACCEPTED)
  public void suspend(@PathVariable("eventId") UUID eventId, @PathVariable("marketId") UUID marketId,
      @Valid @RequestBody MarketStatusChangeRequest body) {
    transition(eventId, marketId, MarketStatus.SUSPENDED, body.reason());
  }
  @PostMapping("/close") @ResponseStatus(HttpStatus.ACCEPTED)
  public void close(@PathVariable("eventId") UUID eventId, @PathVariable("marketId") UUID marketId,
      @Valid @RequestBody MarketStatusChangeRequest body) {
    transition(eventId, marketId, MarketStatus.CLOSED, body.reason());
  }
  @PostMapping("/reopen") @ResponseStatus(HttpStatus.ACCEPTED)
  public void reopen(@PathVariable("eventId") UUID eventId, @PathVariable("marketId") UUID marketId,
      @Valid @RequestBody MarketStatusChangeRequest body) {
    transition(eventId, marketId, MarketStatus.OPEN, body.reason());
  }

  private void transition(UUID eventUuid, UUID marketUuid, MarketStatus next, String reason) {
    EventId eventId = new EventId(eventUuid);
    MarketId marketId = new MarketId(marketUuid);
    MarketStatus previous = cache.getMarketStatus(eventId, marketId).orElse(MarketStatus.OPEN);
    publisher.publishMarketStatusChanged(eventId, marketId, previous, next, reason, clock.instant());
    cache.storeMarketStatus(eventId, marketId, next);
  }

  public record MarketStatusChangeRequest(@NotBlank String reason) {}
}
```

suspend/close/reopen 모두: Redis에서 현재 상태 읽기(없으면 OPEN 기본) →
`MarketStatusChanged`(previous→next) publish → Redis에 새 상태 쓰기. 하류
부수효과(Kafka 전달, consumer)가 비동기라 **202 Accepted**. `@NotBlank reason`
필수.

### FeedOrchestrator 연결 (008 변경)

생성자에 `EventCatalog`를 4번째 인자로 더하고, cache에 쓰는 두 자리(refresh,
lifecycle status 갱신)에서 catalog에도 똑같이 써준다:

```java
public FeedOrchestrator(OddsProvider provider, RedisOddsCache cache,
    OddsFeedPublisher publisher, EventCatalog catalog) { ...; this.catalog = catalog; }

// refresh(): 발견한 이벤트를 cache + catalog 둘 다에
cache.storeEvent(summary);
catalog.put(summary);

// handleLifecycle(): status 갱신본을 cache + catalog 둘 다에
EventSummary updated = new EventSummary(current.eventId(), current.sport(), current.competition(),
    current.homeTeam(), current.awayTeam(), current.scheduledStartAt(), event.status());
cache.storeEvent(updated);
catalog.put(updated);
```

잘못된 커서 400, 단건 200/404, 커서 encode/decode round-trip; odds 200/404;
admin suspend(OPEN→SUSPENDED) publish+store, close 기본 previous=OPEN, reopen
SUSPENDED→OPEN, blank reason 400(Jakarta validation).
