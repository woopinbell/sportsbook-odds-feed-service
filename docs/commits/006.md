# 006 — feat(publisher): Avro Kafka publisher with 1% threshold and eventId key

## 개요

네 개의 wire 토픽을 켠다: `odds.changed`, `market.status.changed`,
`event.lifecycle`, `match.result`. 모든 send는 partition key를 **`eventId`** 로
고정(ADR-0006 — 한 경기의 모든 변경이 같은 partition → 순서 보장). `OddsChanged`
send는 네트워크에 닿기 전 **1% threshold 필터**(repo CLAUDE.md)를 통과한다.

## 작업 순서

1. `AvroSerializer` / `AvroDeserializer` — registry 없는 SpecificRecord codec.
2. `KafkaConfig` — 타입드 `KafkaTemplate<String, SpecificRecord>`.
3. `KafkaTopicsProperties` / `PublishProperties` — 바인딩.
4. `OddsFeedPublisher` — 4 publish 메서드 + threshold.
5. 테스트 — publisher 단위 + Avro codec round-trip.

## 작업 내역

### Avro codec — Schema Registry 없이 (ADR-0014)

```java
public class AvroSerializer<T extends SpecificRecord> implements Serializer<T> {

  static {
    SpecificData.get().addLogicalTypeConversion(new TimeConversions.TimestampMillisConversion());
  }

  @Override
  public byte[] serialize(String topic, T data) {
    if (data == null) return null;
    ByteArrayOutputStream out = new ByteArrayOutputStream();
    try {
      SpecificDatumWriter<T> writer = new SpecificDatumWriter<>(data.getSchema());
      BinaryEncoder encoder = EncoderFactory.get().binaryEncoder(out, null);
      writer.write(data, encoder);
      encoder.flush();
    } catch (IOException e) {
      throw new SerializationException("Failed to serialize Avro record on topic " + topic, e);
    }
    return out.toByteArray();
  }
}
```

Deserializer는 **테스트 전용**(우리가 보낸 걸 검증). 같은 static 변환 등록 +
주입받은 `Schema`로 읽는다:

```java
public class AvroDeserializer<T extends SpecificRecord> implements Deserializer<T> {
  static { SpecificData.get().addLogicalTypeConversion(new TimeConversions.TimestampMillisConversion()); }

  private final Schema schema;
  public AvroDeserializer(Schema schema) { this.schema = schema; }

  @Override
  public T deserialize(String topic, byte[] data) {
    if (data == null) return null;
    try {
      SpecificDatumReader<T> reader = new SpecificDatumReader<>(schema);
      BinaryDecoder decoder = DecoderFactory.get().binaryDecoder(data, null);
      return reader.read(null, decoder);
    } catch (IOException e) {
      throw new SerializationException("Failed to deserialize Avro record on topic " + topic, e);
    }
  }
}
```

- V1은 registry-less(ADR-0014) — 모든 producer/consumer가 단일
  `shared-protocol` 아티팩트를 의존하므로 스키마가 single source of truth.
  Confluent의 "magic byte + schema id" framing 없이 raw Avro binary로 쓴다.
- **static 블록의 `TimestampMillisConversion` 등록이 핵심**. Avro는 기본이
  Joda인데 우리 생성 클래스는 `java.time.Instant`를 쓴다. 이 변환을 전역
  등록 안 하면 `changedAt`/`occurredAt` 필드에서 매 send마다 NPE. 처음
  AvroSerializer를 짤 때 이걸 빼먹어 만났던 함정이라 명시적으로 박았다.
- production consumer는 다른 repo(gateway/betting/settlement)가 각자 가짐.

### KafkaConfig — 타입드 template

```java
@Configuration
public class KafkaConfig {

  @Bean
  public ProducerFactory<String, SpecificRecord> avroProducerFactory(KafkaProperties properties) {
    Map<String, Object> configs = new HashMap<>(properties.buildProducerProperties());
    configs.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
    configs.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, AvroSerializer.class);
    return new DefaultKafkaProducerFactory<>(configs);
  }

  @Bean
  public KafkaTemplate<String, SpecificRecord> avroKafkaTemplate(
      ProducerFactory<String, SpecificRecord> avroProducerFactory) {
    return new KafkaTemplate<>(avroProducerFactory);
  }
}
```

Spring Boot 자동 설정은 `KafkaTemplate<Object,Object>` + ByteArraySerializer를
준다(application.yml 기본값). Avro로 못박는 순간 String key + AvroSerializer 쌍의
타입드 template으로 교체해, **Avro 아닌 send는 컴파일 에러**가 되게 했다.
`buildProducerProperties()`로 bootstrap·acks·idempotence·compression은
application.yml에서 상속하고 serializer만 덮어쓴다. 토픽명과 threshold는 record로
바인딩:

```java
@ConfigurationProperties(prefix = "oddsfeed.kafka.topics")
public record KafkaTopicsProperties(
    String oddsChanged, String marketStatusChanged, String eventLifecycle, String matchResult) {}

@ConfigurationProperties(prefix = "oddsfeed.publish")
public record PublishProperties(BigDecimal oddsChangeThreshold) {}
```

이 두 record를 `ApplicationConfig`의 `@EnableConfigurationProperties` 배열에
`KafkaTopicsProperties.class`·`PublishProperties.class`로 등록한다(005까지의
Mock/Real/Cache properties에 더해).

### OddsFeedPublisher — threshold 필터 + 4 publish

생성자에서 타입드 template·토픽·threshold를 받고, `publishOddsChanged`만
threshold 게이트를 통과한다(나머지 3종은 무조건 send):

```java
@Component
public class OddsFeedPublisher {

  private static final int RELATIVE_SCALE = 6;
  private final KafkaTemplate<String, SpecificRecord> template;
  private final KafkaTopicsProperties topics;
  private final BigDecimal threshold;

  public OddsFeedPublisher(
      KafkaTemplate<String, SpecificRecord> avroKafkaTemplate,
      KafkaTopicsProperties topics, PublishProperties publishProps) {
    this.template = avroKafkaTemplate;
    this.topics = topics;
    this.threshold = publishProps.oddsChangeThreshold();
  }

  public boolean publishOddsChanged(EventId eventId, MarketId marketId, SelectionId selectionId,
      Odds previous, Odds next, Instant changedAt) {
    if (!isSignificantChange(previous, next)) return false;   // 미만 → 아무것도 안 보냄
    OddsChanged event = new OddsChanged(
        eventId.value().toString(), marketId.value().toString(), selectionId.value().toString(),
        previous.decimal().toPlainString(), next.decimal().toPlainString(), changedAt);
    send(topics.oddsChanged(), eventId, event);
    return true;
  }

  boolean isSignificantChange(Odds previous, Odds next) {
    BigDecimal prev = previous.decimal();
    BigDecimal absDiff = next.decimal().subtract(prev).abs();
    BigDecimal relative = absDiff.divide(prev, RELATIVE_SCALE, RoundingMode.HALF_EVEN); // scale 6
    return relative.compareTo(threshold) >= 0;                                          // >= 1%
  }

  public void publishMarketStatusChanged(EventId eventId, MarketId marketId,
      MarketStatus previous, MarketStatus next, String reason, Instant occurredAt) {
    MarketStatusChanged event = new MarketStatusChanged(
        eventId.value().toString(), marketId.value().toString(), previous, next, reason, occurredAt);
    send(topics.marketStatusChanged(), eventId, event);
  }

  public void publishEventLifecycle(EventId eventId, EventLifecycleStatus status,
      Instant scheduledStartAt, Instant occurredAt) {
    // 주의: Avro 생성자 인자 순서는 (id, status, occurredAt, scheduledStartAt).
    EventLifecycle event = new EventLifecycle(eventId.value().toString(), status, occurredAt, scheduledStartAt);
    send(topics.eventLifecycle(), eventId, event);
  }

  public void publishMatchResult(EventId eventId, String score, MatchFinalStatus finalStatus,
      Map<String, String> detail, Instant settledAt) {
    MatchResult event = new MatchResult(eventId.value().toString(), score, finalStatus, detail, settledAt);
    send(topics.matchResult(), eventId, event);
  }

  private void send(String topic, EventId eventId, SpecificRecord payload) {
    template.send(topic, eventId.value().toString(), payload);   // partition key = eventId
  }
}
```

- `publishOddsChanged`는 **boolean을 반환** — orchestrator가 threshold 산수를
  다시 안 하고도 필터 비율을 메트릭으로 노출할 수 있게.
- 경계: `2.00 → 2.02`는 정확히 0.0100이라 publish, `2.00 → 2.019`는 0.0095로
  반올림돼 publish 안 함. `compareTo(threshold) >= 0`이라 **정확히 1%는 통과**.

> **C/C++ 메모.** 금융 계산에 `double`을 쓰면 0.01이 정확히 표현 안 돼
> 경계 비교가 어긋난다. Java `BigDecimal`은 임의 정밀 십진수 — `divide`에
> scale·RoundingMode를 명시해야 한다(안 하면 무한소수에서 예외). C++엔 표준
> BigDecimal이 없어 보통 정수 minor unit이나 외부 라이브러리를 쓰는 것과 대비.

### partition key = eventId

`send`가 네 토픽 전부 `eventId.value().toString()`을 key로 쓴다. 한 경기의 odds
변경·market status·lifecycle·result가 같은 partition에 떨어져 consumer가
**전순서(total order)** 로 본다.

## 결과

테스트 통과(누적 49). `OddsFeedPublisherTest`(7): threshold 미만 억제, 초과
payload+토픽, 정확히 1% 경계, **네 토픽 전부 partition key=eventId**(ADR-0006
잠금), MarketStatusChanged reason, EventLifecycle 두 timestamp, MatchResult
score+detail. `AvroSerializerTest`(4): OddsChanged/MarketStatusChanged/
EventLifecycle binary round-trip + null 입력.

**Testcontainers Kafka end-to-end 테스트는 짰다가 들어냈다** —
`confluentinc/cp-kafka` 이미지가 ARM macOS에서 기동 실패. Avro codec 테스트 +
publisher 단위 테스트가 양쪽(codec 정확성 + send 형태)을 Docker 의존 없이 덮는다.
시스템 전체 Kafka 검증은 orchestration repo의 e2e(Phase 5).

## 요약

- registry 없는 Avro codec, static 블록에 Instant 변환 등록(NPE 방지).
- 타입드 `KafkaTemplate<String,SpecificRecord>` — 비-Avro send 컴파일 차단.
- threshold는 `BigDecimal` scale-6 비교, 정확히 1% 통과, boolean 반환.
- 4 토픽 전부 partition key=eventId.
