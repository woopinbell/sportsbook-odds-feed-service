# 002 — feat(provider): MockOddsProvider with deterministic event simulator

## 개요

`OddsProvider`의 **기본 구현체**. `@Profile("mock")`로 활성화되고, 외부 키 없이
로컬에서 전체 이벤트 lifecycle을 굴린다. 부하 테스트·통합 테스트·카오스
시나리오가 전부 이 위에서 돈다(ADR-0010 — mock 1차). 핵심은 **결정론
(deterministic)** — 같은 seed면 같은 결과라 테스트가 재현 가능하다.

## 작업 순서

1. `ApplicationConfig` — `@EnableScheduling` + `Clock` @Bean.
2. `MockProperties` — `oddsfeed.mock.*` 바인딩.
3. `OddsSimulator` — 순수 가격 random walk (Spring 무관).
4. `MockOddsProvider` — lifecycle 진행 + odds emit + 결과 합성.
5. 단위 테스트 — lifecycle, emit, 결과, 시뮬레이터 수렴/클램핑/결정성.

## 작업 내역

### OddsSimulator — 순수 함수로 분리

가격 모델을 Spring·스레드와 무관한 순수 클래스로 떼냈다. 테스트가 컨테이너 없이
바로 돈다.

```java
// fair odds = 1/p. 곱셈 노이즈로 흔든 뒤 fair 쪽으로 일부 평균회귀.
public static Odds nextOdds(Odds current, double impliedProbability, Random rng) {
  double fair = 1.0 / impliedProbability;
  double cur = current.decimal().doubleValue();
  double noisy = cur * (1.0 + rng.nextGaussian() * NOISE_STDDEV);   // 2%
  double next = noisy * (1.0 - MEAN_REVERSION) + fair * MEAN_REVERSION; // 10%
  double clamped = Math.max(MIN_ODDS, Math.min(MAX_ODDS, next));     // [1.01, 100]
  return Odds.ofDecimal(BigDecimal.valueOf(clamped).setScale(Odds.SCALE, HALF_EVEN));
}
```

- **평균회귀(mean reversion)** 가 핵심. 순수 random walk는 발산하거나 1.0 밑으로
  내려가 `Odds`(>= 1.0만 허용)가 예외를 던진다. fair 쪽으로 10%씩 당기고
  `[1.01, 100]`으로 clamp해서 항상 유효한 odds만 나오게 했다.
- base 확률: HOME 0.45 / DRAW 0.25 / AWAY 0.30 (합 1.0, **margin 0** — V1).

### MockOddsProvider — 시간 진행 + emit

```java
@Scheduled(fixedRateString = "${oddsfeed.mock.tick-interval-ms:500}")
void scheduledTick() { tick(clock.instant()); }

void tick(Instant now) {            // package-private — 테스트가 직접 호출
  for (MockEvent event : events.values()) advance(event, now);
}
```

- `@PostConstruct`에서 EPL 3경기를 seed. 이후 `@Scheduled` tick이
  SCHEDULED → IN_PLAY → FINISHED로 진행시킨다.
- 변경은 이벤트별 multicast `Sinks.Many`로 흘려 orchestrator가 구독.
- **시간 scale**: `oddsfeed.mock.minutes-per-second`로 mock시간을 압축.
  기본 1(90분 경기가 90초), 부하 테스트는 60(roster가 몇 초마다 순환).

> **테스트 가능성.** `Clock`을 주입한다(@Bean은 system UTC, 테스트는
> `Clock.fixed`). `@Scheduled` 메서드는 `clock.instant()`를 읽지만, 테스트는
> 스케줄러를 우회하고 package-private `tick(Instant)`를 직접 부른다. 그래서
> `@EnableScheduling`이 테스트 클래스패스에 안 들어와도 된다. C/C++에서 시간
> 의존 코드를 테스트할 때 `now()`를 함수 포인터로 주입하던 것과 같은 발상.

### V1 scope를 좁게

- 이벤트당 `MATCH_RESULT_1X2` 마켓 1개 + selection 3개(HOME/DRAW/AWAY).
- 시나리오 회전·마켓 suspend·late goal은 **없음** — 다음 커밋(003)의
  `MockScenario`. 단 `MockProperties.Scenarios`에 auto-rotate 플래그는 미리
  넣어둬서 003이 이 클래스 모양을 안 건드리게 했다.

## 결과

19개 테스트 통과. **여기서 산수 버그를 하나 잡았다** — `toRealDuration`이 mock분
→ 실초 변환에서 60을 한 번 더 곱해 90분 경기가 5400초가 됐다.
`tickProgressesEventThroughFullLifecycle`가 FINISHED 도달 실패로 빨갛게 떠서
즉시 발견·수정. 단위 테스트가 "조용한 산수 실수"를 잡은 정확한 사례다.

## 요약

- `OddsSimulator`: fair odds 중심 평균회귀 random walk + clamp → 항상 유효 odds.
- `MockOddsProvider`: `@Scheduled` tick으로 lifecycle 진행, 이벤트별 Sink emit.
- `Clock` 주입 + package-private `tick`으로 스케줄러 없이 테스트.
- 시간 scale로 부하 테스트 시 빠른 lifecycle 재현.

- **L2**: `Clock` 주입 테스트 전략, 시간 scale 의미, V1 마켓 1종 제약.
- **L3**: **평균회귀가 왜 필요한가**(발산·`Odds<1.0` 예외 방지),
  **순수 `OddsSimulator` 분리의 테스트 이점**, **결정론적 seed**. mock 산수
  버그를 테스트가 잡은 일화는 "테스트가 실제로 일했다"는 좋은 면접 소재.
