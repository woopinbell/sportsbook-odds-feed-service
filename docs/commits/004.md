# 004 — feat(provider): TheOddsApiProvider for the real upstream

## 개요

`OddsProvider`의 **real 구현**(`@Profile("real")`). The Odds API v4를 WebClient로
폴링한다. 무료 tier(500 req/월, 5 req/분 권장)를 chatty한 poll 루프가 태워버리지
않도록 **sliding-window RateLimiter**(분당)와 **Redis 월간 quota 카운터**가 둘 다
바깥 호출을 막는다. mock과 같은 인터페이스라 orchestrator는 profile을 모른다.

## 작업 순서

1. `pom.xml`에 WireMock(test) 추가.
2. `RealProperties` — `oddsfeed.real.*` 바인딩.
3. `TheOddsApiDtos` — v4 응답 wire 모델.
4. `RateLimiter` (sliding window) + `QuotaCounter` 인터페이스 + `RedisQuotaCounter`.
5. `TheOddsApiProvider` — poll + diff + emit.
6. `TheOddsApiClientConfig` — WebClient·RateLimiter @Bean.
7. 테스트 — RateLimiter 단위 + WireMock 기반 provider.

## 작업 내역

### wire 모델 — snake_case → camelCase

업스트림 v4 응답을 한 파일에 record 4중첩으로. 클래스 레벨 `@JsonNaming`으로 Java
필드는 camelCase, Jackson은 업스트림이 보내는 snake_case(`sport_key`,
`commence_time`) 키를 읽는다:

```java
public final class TheOddsApiDtos {
  private TheOddsApiDtos() {}

  @JsonNaming(SnakeCaseStrategy.class)
  public record Event(String id, String sportKey, String sportTitle,
      Instant commenceTime, String homeTeam, String awayTeam, List<Bookmaker> bookmakers) {}

  @JsonNaming(SnakeCaseStrategy.class)
  public record Bookmaker(String key, String title, Instant lastUpdate, List<Market> markets) {}

  @JsonNaming(SnakeCaseStrategy.class)
  public record Market(String key, Instant lastUpdate, List<Outcome> outcomes) {}

  @JsonNaming(SnakeCaseStrategy.class)
  public record Outcome(String name, BigDecimal price) {}
}
```

DTO는 real 패키지를 안 벗어나고 — adapter 경계에서 `EventSummary`·
`ProviderEvent.OddsUpdated`로 변환된다.

### poll + diff

`@Scheduled`가 sport마다 `pollSport`를 부르고, 테스트는 스케줄러를 우회해
package-private `pollSport`를 직접 부른다. `fetchEvents`가 두 게이트(rate limit →
quota)를 통과한 뒤에만 업스트림을 친다:

```java
@Scheduled(fixedRateString = "${oddsfeed.real.poll-interval-seconds:60}", timeUnit = TimeUnit.SECONDS)
void scheduledPoll() {
  for (Sport sport : Sport.values()) pollSport(sport);
}

void pollSport(Sport sport) {
  String sportKey = sportToApiKey(sport);
  if (sportKey == null) return;
  List<TheOddsApiDtos.Event> raw = fetchEvents(sportKey);
  if (raw == null) return;                          // rate-limited / quota-blown / error
  for (TheOddsApiDtos.Event dto : raw) {
    EventId eventId = deriveEventId(dto.id());
    Snapshot next = toSnapshot(dto, eventId);
    Snapshot previous = lastSeen.get(eventId);
    if (previous != null) emitDiff(eventId, previous, next);
    lastSeen.put(eventId, next);
  }
}

private List<TheOddsApiDtos.Event> fetchEvents(String sportKey) {
  if (!rateLimiter.tryAcquire()) return null;       // 분당 게이트 — short-circuit
  long used = quotaCounter.increment();
  if (used > props.monthlyQuota()) return null;     // 월간 게이트
  try {
    TheOddsApiDtos.Event[] response = client.get()
        .uri(b -> b.path("/sports/{key}/odds")
            .queryParam("apiKey", props.apiKey())
            .queryParam("regions", "uk")
            .queryParam("markets", H2H_MARKET_KEY)   // "h2h" = MATCH_RESULT_1X2
            .queryParam("oddsFormat", "decimal")
            .build(sportKey))
        .retrieve().bodyToMono(TheOddsApiDtos.Event[].class).block(FETCH_TIMEOUT);
    return response == null ? List.of() : List.of(response);
  } catch (RuntimeException ex) {
    return null;                                      // 실패는 null → 이번 poll skip
  }
}
```

- 업스트림은 움직임 여부와 무관하게 매 poll마다 전체 응답을 준다. 그래서
  `(marketKey, outcomeName) → Odds` 스냅샷을 직전과 **diff**해서 바뀐 가격만
  `OddsUpdated`로 emit한다. 이게 publisher를 firehosing하지 않는 유일한 필터.
- `getMatchResult`는 V1에서 `Optional.empty()` — `/scores` 연동은 V2(클래스
  Javadoc에 명시, TODO 아님).

스냅샷 빌드와 diff — **첫 bookmaker의 h2h만** 읽고(V1), diff는 outcomeName 정렬로
결정론적 순서를 보장한다:

```java
private Snapshot toSnapshot(TheOddsApiDtos.Event dto, EventId eventId) {
  Map<SelectionKey, Odds> prices = new HashMap<>();
  MarketId marketId = deriveMarketId(eventId, H2H_MARKET_KEY);
  if (dto.bookmakers() == null || dto.bookmakers().isEmpty())
    return new Snapshot(prices, marketId, dto.commenceTime());
  TheOddsApiDtos.Bookmaker book = dto.bookmakers().get(0);
  for (TheOddsApiDtos.Market market : book.markets()) {
    if (!H2H_MARKET_KEY.equals(market.key())) continue;
    for (TheOddsApiDtos.Outcome outcome : market.outcomes())
      prices.put(new SelectionKey(market.key(), outcome.name()), Odds.ofDecimal(outcome.price()));
  }
  return new Snapshot(prices, marketId, book.lastUpdate());
}

private void emitDiff(EventId eventId, Snapshot previous, Snapshot next) {
  Sinks.Many<ProviderEvent> sink = streams.get(eventId);
  if (sink == null) return;
  next.prices().entrySet().stream()
      .sorted(Comparator.comparing(e -> e.getKey().outcomeName()))   // 안정 순서
      .forEach(entry -> {
        SelectionKey key = entry.getKey();
        Odds newOdds = entry.getValue();
        Odds prevOdds = previous.prices().get(key);
        if (prevOdds == null || !prevOdds.equals(newOdds)) {         // 신규 또는 변경만
          Odds emittedPrevious = prevOdds == null ? newOdds : prevOdds;
          sink.tryEmitNext(new ProviderEvent.OddsUpdated(
              eventId, next.marketId(), deriveSelectionId(eventId, key),
              emittedPrevious, newOdds, next.observedAt()));
        }
      });
}

record Snapshot(Map<SelectionKey, Odds> prices, MarketId marketId, Instant observedAt) {}
record SelectionKey(String marketKey, String outcomeName) {}
```

### ID 유도 — opaque string → 결정론적 UUID

The Odds API는 UUID가 아닌 불투명 문자열 id를 쓴다. `UUID.nameUUIDFromBytes`(타입
5 name-based UUID)로 **결정론적**으로 UUID를 만들어, 별도 lookup 테이블 없이도
poll마다 같은 `EventId`/`MarketId`/`SelectionId`가 나오게 했다:

```java
static EventId deriveEventId(String upstreamId) {
  return new EventId(UUID.nameUUIDFromBytes(upstreamId.getBytes(StandardCharsets.UTF_8)));
}
static MarketId deriveMarketId(EventId eventId, String marketKey) {
  return new MarketId(UUID.nameUUIDFromBytes(
      (eventId.value() + ":" + marketKey).getBytes(StandardCharsets.UTF_8)));
}
static SelectionId deriveSelectionId(EventId eventId, SelectionKey key) {
  return new SelectionId(UUID.nameUUIDFromBytes(
      (eventId.value() + ":" + key.marketKey() + ":" + key.outcomeName()).getBytes(StandardCharsets.UTF_8)));
}
```

> **C/C++ 메모.** `nameUUIDFromBytes`는 입력 바이트의 MD5 해시를 UUID 포맷에
> 끼우는 결정론 함수다 — 같은 입력 → 항상 같은 UUID. 외부 키를 내부 타입에
> 안정적으로 매핑할 때 흔히 쓴다.

### rate limit + quota

`RateLimiter`는 최근 타임스탬프의 sliding 1분 윈도우. 캡 N(무료 tier ≤5)에 대해
O(N)이라 deque 하나면 충분(token bucket은 과설계). `Clock` 주입으로 테스트가
sleep 없이 시간을 진행:

```java
public final class RateLimiter {
  private static final Duration WINDOW = Duration.ofMinutes(1);
  private final int maxRequestsPerWindow;
  private final Clock clock;
  private final Deque<Instant> recent = new ArrayDeque<>();

  public RateLimiter(int maxRequestsPerWindow, Clock clock) {
    this.maxRequestsPerWindow = maxRequestsPerWindow;
    this.clock = clock;
  }

  public synchronized boolean tryAcquire() {
    Instant now = clock.instant();
    Instant cutoff = now.minus(WINDOW);
    while (!recent.isEmpty() && recent.peekFirst().isBefore(cutoff)) {
      recent.pollFirst();                  // 윈도우 밖 타임스탬프 evict
    }
    if (recent.size() >= maxRequestsPerWindow) return false;
    recent.addLast(now);
    return true;
  }
}
```

`QuotaCounter`는 **인터페이스** — 여기선 추상화가 정당하다(테스트는 in-memory,
production은 Redis, **실제 두 구현이 존재**). `RedisQuotaCounter`는 월별 키에 35일
TTL, `INCR`은 atomic:

```java
public interface QuotaCounter {
  long increment();   // 이번 달 카운터 atomic 증가, 새 값 반환
  long current();
}

@Component
@Profile("real")
public class RedisQuotaCounter implements QuotaCounter {
  private static final String KEY_PREFIX = "oddsfeed:thequota:";
  private static final DateTimeFormatter MONTH_FORMAT = DateTimeFormatter.ofPattern("yyyy-MM");
  private static final Duration TTL = Duration.ofDays(35);   // 지난달 자동 만료

  @Override public long increment() {
    String key = currentKey();
    Long incremented = redis.opsForValue().increment(key);    // atomic INCR
    redis.expire(key, TTL);
    return incremented == null ? 0L : incremented;
  }
  private String currentKey() {
    return KEY_PREFIX + MONTH_FORMAT.format(clock.instant().atZone(ZoneOffset.UTC));
  }
}
```

> **003과 대비.** 003에선 추상화를 일부러 안 만들었는데(YAGNI) 여기선
> `QuotaCounter`를 인터페이스로 뽑았다. 차이는 **실제로 두 구현이 존재**하기
> 때문 — 테스트용 in-memory와 운영용 Redis. "두 번째 구현이 있을 때 추출한다"는
> 003의 기준을 그대로 적용한 것.

### 바인딩 + WebClient @Bean

`RealProperties`는 `oddsfeed.real.*`을 바인딩하고(누락 시 real profile에서 context
init이 fail-fast), `TheOddsApiClientConfig`가 WebClient·RateLimiter를 @Bean으로
(Redis quota counter는 component scan으로 자기 등록):

```java
@ConfigurationProperties(prefix = "oddsfeed.real")
public record RealProperties(String apiKey, String baseUrl, List<String> sportKeys,
    RateLimit rateLimit, int monthlyQuota, int pollIntervalSeconds) {
  public record RateLimit(int maxRequestsPerMinute) {}
}

@Configuration
@Profile("real")
public class TheOddsApiClientConfig {
  @Bean public WebClient theOddsWebClient(RealProperties props) {
    return WebClient.builder().baseUrl(props.baseUrl()).build();
  }
  @Bean public RateLimiter theOddsRateLimiter(RealProperties props, Clock clock) {
    return new RateLimiter(props.rateLimit().maxRequestsPerMinute(), clock);
  }
}
```

`ApplicationConfig`에 `RealProperties.class`를 등록(`@EnableConfigurationProperties`
배열 확장)하고, `pom.xml`엔 WireMock(test)을 더해 실제 quota를 안 태우고 WebClient·
DTO 매핑을 검증한다.

## 결과

테스트 통과(누적 33). WireMock으로 `/sports/soccer_epl/odds`를 stub해서: EventSummary
매핑, fetch당 quota 증가, **바뀐 가격만 emit·안 바뀐 건 조용**(맨유 1.85→1.90,
첼시 4.20→4.00, Draw 3.60 고정 → 정확히 2개 emit), rate-limit 소진 시 업스트림
호출 안 함(quota 미증가)을 검증.

WireMock stub 초기화 API(`removeStubsByMetadata`)의 시그니처를 처음에 잘못 써서
컴파일 에러 → `resetMappings()`로 교체.
