# 004 — feat(provider): TheOddsApiProvider for the real upstream

## 개요

`OddsProvider`의 **real 구현**(`@Profile("real")`). The Odds API v4를 WebClient로
폴링한다. 무료 tier(500 req/월, 5 req/분 권장)를 chatty한 poll 루프가 태워버리지
않도록 **sliding-window RateLimiter**(분당)와 **Redis 월간 quota 카운터**가 둘 다
바깥 호출을 막는다. mock과 같은 인터페이스라 orchestrator는 profile을 모른다.

## 작업 순서

1. `pom.xml`에 WireMock(test) 추가.
2. `RealProperties` — `oddsfeed.real.*` 바인딩.
3. `TheOddsApiDtos` — v4 응답 wire 모델.
4. `RateLimiter` (sliding window) + `QuotaCounter` 인터페이스 + `RedisQuotaCounter`.
5. `TheOddsApiProvider` — poll + diff + emit.
6. `TheOddsApiClientConfig` — WebClient·RateLimiter @Bean.
7. 테스트 — RateLimiter 단위 + WireMock 기반 provider.

## 작업 내역

### wire 모델 — snake_case → camelCase

```java
@JsonNaming(SnakeCaseStrategy.class)
public record Event(String id, String sportKey, String sportTitle,
                    Instant commenceTime, String homeTeam, String awayTeam,
                    List<Bookmaker> bookmakers) {}
```

클래스 레벨 `@JsonNaming`으로 Java 필드는 camelCase, Jackson은 업스트림이 보내는
snake_case(`sport_key`, `commence_time`) 키를 읽는다. DTO는 real 패키지를 안
벗어나고 — adapter 경계에서 `EventSummary`·`ProviderEvent.OddsUpdated`로 변환된다.

### poll + diff

```java
void pollSport(Sport sport) {           // package-private — 테스트가 스케줄러 우회
  ...
  for (Event dto : raw) {
    EventId eventId = deriveEventId(dto.id());
    Snapshot next = toSnapshot(dto, eventId);
    Snapshot previous = lastSeen.get(eventId);
    if (previous != null) emitDiff(eventId, previous, next);
    lastSeen.put(eventId, next);
  }
}
```

- 업스트림은 움직임 여부와 무관하게 매 poll마다 전체 응답을 준다. 그래서
  `(marketKey, outcomeName) → Odds` 스냅샷을 직전과 **diff**해서 바뀐 가격만
  `OddsUpdated`로 emit한다. 이게 publisher를 firehosing하지 않는 유일한 필터.
- `getMatchResult`는 V1에서 `Optional.empty()` — `/scores` 연동은 V2(클래스
  Javadoc에 명시, TODO 아님).

### ID 유도 — opaque string → 결정론적 UUID

```java
static EventId deriveEventId(String upstreamId) {
  return new EventId(UUID.nameUUIDFromBytes(upstreamId.getBytes(UTF_8)));
}
```

The Odds API는 UUID가 아닌 불투명 문자열 id를 쓴다. `UUID.nameUUIDFromBytes`(타입
5 name-based UUID)로 **결정론적**으로 UUID를 만들어, 별도 lookup 테이블 없이도
poll마다 같은 `EventId`/`MarketId`/`SelectionId`가 나오게 했다.

> **C/C++ 메모.** `nameUUIDFromBytes`는 입력 바이트의 MD5 해시를 UUID 포맷에
> 끼우는 결정론 함수다 — 같은 입력 → 항상 같은 UUID. 외부 키를 내부 타입에
> 안정적으로 매핑할 때 흔히 쓴다.

### rate limit + quota

- `RateLimiter` — 최근 타임스탬프의 sliding 1분 윈도우. 캡 N(무료 tier ≤5)에
  대해 O(N), 사소함. `Clock` 주입으로 테스트가 sleep 없이 시간을 진행.
- `QuotaCounter`는 **인터페이스**(여기선 추상화가 정당 — 테스트는 in-memory,
  production은 Redis). `RedisQuotaCounter`는 `oddsfeed:thequota:YYYY-MM` 키에
  35일 TTL, `INCR`은 atomic.

> **003과 대비.** 003에선 추상화를 일부러 안 만들었는데(YAGNI) 여기선
> `QuotaCounter`를 인터페이스로 뽑았다. 차이는 **실제로 두 구현이 존재**하기
> 때문 — 테스트용 in-memory와 운영용 Redis. "두 번째 구현이 있을 때 추출한다"는
> 003의 기준을 그대로 적용한 것.

## 결과

테스트 통과(누적 33). WireMock으로 `/sports/soccer_epl/odds`를 stub해서: EventSummary
매핑, fetch당 quota 증가, **바뀐 가격만 emit·안 바뀐 건 조용**(맨유 1.85→1.90,
첼시 4.20→4.00, Draw 3.60 고정 → 정확히 2개 emit), rate-limit 소진 시 업스트림
호출 안 함(quota 미증가)을 검증.

WireMock stub 초기화 API(`removeStubsByMetadata`)의 시그니처를 처음에 잘못 써서
컴파일 에러 → `resetMappings()`로 교체.

## 요약

- WebClient + The Odds API v4, snake_case wire 모델.
- poll + 스냅샷 diff로 바뀐 가격만 emit.
- 결정론 UUID 유도로 lookup 테이블 회피.
- RateLimiter(분당) + Redis quota(월간) 이중 게이트.
- `getMatchResult`는 V1 empty (V2 `/scores`).

- **L3**: **결정론 UUID 유도**(왜 lookup 없이 되나), **`QuotaCounter`는
  인터페이스인데 003의 scenario는 왜 아닌가**(추상화 기준의 일관성),
  **diff가 유일한 firehose 방지선**. "외부 API rate limit을 어떻게 다뤘나" 단골.
