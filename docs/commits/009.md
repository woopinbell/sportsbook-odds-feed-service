## 개요

## 작업 내역

### 측정 형태

`@Tag("load")`로 CI 제외 가능. 측정 본문은 warm-up 1건 → `EVENT_COUNT` 순차 send
→ flush까지 wall-time → events/sec 계산 → 하한 assertion:

```java
@Tag("load")   // CI는 -DexcludedGroups=load 로 제외
class KafkaPublishThroughputTest {

  private static final int EVENT_COUNT = 100_000;
  private static final KafkaTopicsProperties TOPICS =
      new KafkaTopicsProperties("odds.changed", "market.status.changed", "event.lifecycle", "match.result");

  @Test
  void publishesAtLeastTenKEventsPerSecond() {
    EventId eventId = new EventId(UUID.randomUUID());
    MarketId marketId = new MarketId(UUID.randomUUID());
    SelectionId selectionId = new SelectionId(UUID.randomUUID());
    Odds previous = Odds.ofDecimal("2.00");
    Odds next = Odds.ofDecimal("2.10");                 // 5% 이동 — 매 send가 1% threshold 통과
    Instant when = Instant.parse("2026-05-28T10:00:00Z");

    // Warm-up: 첫 send는 producer-init 비용을 문다. 버린다.
    publisher.publishOddsChanged(eventId, marketId, selectionId, previous, next, when);

    Instant start = Instant.now();
    for (int i = 0; i < EVENT_COUNT; i++) {
      publisher.publishOddsChanged(eventId, marketId, selectionId, previous, next, when);
    }
    producerFactory.createProducer().flush();           // 배치 비우고 끝
    Duration elapsed = Duration.between(start, Instant.now());

    double tps = EVENT_COUNT / (elapsed.toMillis() / 1000.0);
    assertThat(tps).isGreaterThan(10_000.0);            // flush-per-message 같은 회귀 방지 하한
  }
}
```

- OddsChanged 10만 건, decimal 2.0000 → 2.1000이라 **매 send가 1% threshold를
  넘김**(안 그러면 필터가 측정을 지배).
- 단일 스레드 순차, warm-up 1건(producer-init 비용 지불), 그 뒤 첫 카운트 send부터
  flush까지 wall-time.
- assertion에 **하한 10k/sec** — flush-per-message나 우발적 sync send 같은
  명백한 회귀를 잡는 안전망. 실제 숫자는 결과 파일로.

### broker 선택 — EmbeddedKafkaZKBroker, Testcontainers 아님

`@BeforeAll`에서 in-process broker를 띄우고 producer를 직접 튜닝한다.
`EmbeddedKafkaBroker`는 abstract라 구상 클래스 `EmbeddedKafkaZKBroker`를 써야
했다:

```java
@BeforeAll
static void start() {
  broker = new EmbeddedKafkaZKBroker(1, true, 3,           // 1 broker, 토픽당 3 partition
      TOPICS.oddsChanged(), TOPICS.marketStatusChanged(), TOPICS.eventLifecycle(), TOPICS.matchResult());
  broker.afterPropertiesSet();

  Map<String, Object> producerProps = new HashMap<>();
  producerProps.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, broker.getBrokersAsString());
  producerProps.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
  producerProps.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, AvroSerializer.class);
  producerProps.put(ProducerConfig.LINGER_MS_CONFIG, 5);
  producerProps.put(ProducerConfig.BATCH_SIZE_CONFIG, 32 * 1024);
  producerProps.put(ProducerConfig.COMPRESSION_TYPE_CONFIG, "lz4");
  producerProps.put(ProducerConfig.ACKS_CONFIG, "1");     // acks=all은 내구성 게이트지 측정 대상 아님
  producerFactory = new DefaultKafkaProducerFactory<>(producerProps);
  publisher = new OddsFeedPublisher(
      new KafkaTemplate<>(producerFactory), TOPICS, new PublishProperties(new BigDecimal("0.01")));
}
```

publisher 커밋(006)에서 `confluentinc/cp-kafka:7.5.0` 이미지가 ARM macOS에서
기동 실패했다. 이미지와 싸우는 대신 문서화된 fallback을 택했다 —
`spring-kafka-test`의 `EmbeddedKafkaZKBroker`(in-process). 같은 Kafka 브로커
코드를 in-process로 돌리므로 publisher 측 측정엔 유효하지만, **수치는 JVM
ceiling** — 실제 TCP·네트워크·복제가 있는 배포는 더 낮다(단일 broker ~50-150k/s
추정).

- **producer 튜닝**: `acks=1`, `linger.ms=5`, `batch.size=32K`, `lz4`. `acks=all`은
  처리량을 절반 이하로 떨구는데, 그건 **내구성**을 게이트하지 직렬화/배치
  경로를 게이트하지 않아서 측정 대상이 아니다. 그래서 acks=1.

약 **290k events/sec**로 안정 — 목표 10만의 **약 3배**. `results/2026-05-28/
BASELINE.md`(실행별 표)와 `results/BEST.md`(누적 최고)에 박제.

> **이 커밋의 정직한 범위.** 커밋 제목은 "load baseline"이지만 실제로 잰 건
> publisher → in-process broker 단일 지표다. end-to-end latency(mock → Kafka p99)
> 와 HTTP read p99는 **측정 안 함** — 전자는 mock·broker가 별도 프로세스여야
> 의미가 있어 orchestration Phase 5 e2e로, 후자는 실행 인스턴스 + k6 바이너리가
> 필요해 미실행. `odds_read.js`는 골격만 있다(현재 read API가 market/selection id를
> 직접 노출 안 해 events 목록 경로만 구동).
