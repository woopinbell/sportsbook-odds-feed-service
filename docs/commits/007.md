# 007 — feat(orchestrator): wire provider, cache, and publisher together

## 개요

`FeedOrchestrator`는 이 서비스의 세 부품(provider·cache·publisher)이 다 존재함을
아는 **유일한** 조립 지점이다. 활성 `OddsProvider`(profile에 따라 mock 또는
real)에서 `EventSummary`를 끌어와, 이벤트별 `ProviderEvent` 스트림을 구독하고,
각 이벤트를 알맞은 cache + publish 경로로 보낸다. **profile 선택이 이 클래스를
넘어 전파되지 않는다** — cache도 publisher도 자기가 mock을 섬기는지 real을
섬기는지 모른다. adapter 패턴(001)의 배당금이 여기서 실현된다.

## 작업 순서

1. 4-인자 생성자(provider, cache, publisher, + 008에서 catalog).
2. discovery 루프 — `@PostConstruct` + `@Scheduled refresh()`.
3. 이벤트별 `streamEvents` 구독(중복 방지).
4. `dispatch` — sealed `ProviderEvent` 분기 → cache+publish.
5. 7개 mock 기반 테스트.

## 작업 내역

### discovery 루프

생성자로 세 부품 + 구독 추적 맵을 받고, `@PostConstruct`가 첫 HTTP 요청 전에
캐시를 채운다. `@Scheduled refresh`가 sport별 이벤트를 훑으며 메타데이터를
저장하고 신규 이벤트만 구독한다:

```java
@Component
public class FeedOrchestrator {

  private final OddsProvider provider;
  private final RedisOddsCache cache;
  private final OddsFeedPublisher publisher;
  private final Map<EventId, Disposable> subscriptions = new ConcurrentHashMap<>();

  public FeedOrchestrator(OddsProvider provider, RedisOddsCache cache, OddsFeedPublisher publisher) {
    this.provider = provider;
    this.cache = cache;
    this.publisher = publisher;
  }

  @PostConstruct
  void start() {
    refresh();                       // 첫 HTTP 요청 전에 캐시 채움
  }

  @PreDestroy
  void stop() {
    subscriptions.values().forEach(Disposable::dispose);   // 깨끗한 shutdown
    subscriptions.clear();
  }

  @Scheduled(fixedRateString = "${oddsfeed.orchestrator.refresh-interval-seconds:30}",
             timeUnit = TimeUnit.SECONDS)
  void refresh() {
    for (Sport sport : Sport.values()) {
      for (EventSummary summary : provider.listEvents(sport)) {
        cache.storeEvent(summary);
        subscriptions.computeIfAbsent(summary.eventId(), this::subscribe);   // 이벤트당 정확히 1회
      }
    }
  }

  private Disposable subscribe(EventId eventId) {
    return provider.streamEvents(eventId).subscribe(event -> dispatch(eventId, event));
  }
```

`computeIfAbsent`로 이벤트당 구독을 **정확히 한 번**만(여러 refresh 사이클에도).
`@PreDestroy`로 종료 시 구독 해제.

> **C/C++ 메모.** `@PostConstruct`/`@PreDestroy`는 Spring 빈 생명주기 훅 —
> 생성자 주입이 끝난 뒤(`@PostConstruct`)와 컨테이너 종료 전(`@PreDestroy`)
> 호출된다. C++의 RAII(생성자/소멸자)와 달리 Spring이 시점을 관리한다.

### cache vs publish 분리 (박아둘 정책 결정)

세 핸들러 모두 **publish 먼저, cache 나중**. odds·market status는 새 값을
그대로, lifecycle은 현재 `EventSummary`에서 status만 바꿔 재저장:

```java
private void handleOdds(ProviderEvent.OddsUpdated event) {
  publisher.publishOddsChanged(event.eventId(), event.marketId(), event.selectionId(),
      event.previousOdds(), event.newOdds(), event.occurredAt());   // 1% 미만이면 내부에서 skip
  cache.storeOdds(event.eventId(), event.marketId(), event.selectionId(), event.newOdds()); // 무조건
}

private void handleMarketStatus(ProviderEvent.MarketStatusUpdated event) {
  publisher.publishMarketStatusChanged(event.eventId(), event.marketId(),
      event.previousStatus(), event.newStatus(), event.reason(), event.occurredAt());
  cache.storeMarketStatus(event.eventId(), event.marketId(), event.newStatus());
}

private void handleLifecycle(ProviderEvent.LifecycleUpdated event) {
  publisher.publishEventLifecycle(
      event.eventId(), event.status(), event.scheduledStartAt(), event.occurredAt());
  cache.getEvent(event.eventId()).ifPresent(current ->       // kickoff·팀명 보존, status만 교체
      cache.storeEvent(new EventSummary(
          current.eventId(), current.sport(), current.competition(),
          current.homeTeam(), current.awayTeam(), current.scheduledStartAt(), event.status())));

  if (event.status() == EventLifecycleStatus.FINISHED) {     // match-result 분기 (아래)
    Optional<MatchOutcome> outcome = provider.getMatchResult(event.eventId());
    outcome.ifPresent(o -> publisher.publishMatchResult(
        o.eventId(), o.score(), o.finalStatus(), o.detail(), o.settledAt()));
  }
}
```

- **cache 갱신은 모든 이벤트에 무조건.** 1% threshold는 publisher에만 산다 —
  그건 Kafka 노이즈를 줄이는 거지 "현재 odds가 뭔지"를 바꾸는 게 아니다.
  betting-service는 슬립 검증에 **가장 신선한 odds**를 읽어야 하므로 wire
  메시지가 안 나갔어도 cache는 canonical을 유지. 안 그러면 sub-1% drift가
  쌓여 cache가 현실과 어긋난다.
- lifecycle 전이: cache에서 현재 `EventSummary`를 꺼내 status만 바꿔 다시 쓴다
  (kickoff·팀명은 안 변하니 provider 재조회 불필요).

### dispatch — Java 17이라 instanceof 체인

```java
void dispatch(EventId eventId, ProviderEvent event) {
  if (event instanceof ProviderEvent.OddsUpdated odds) {
    handleOdds(odds);
  } else if (event instanceof ProviderEvent.MarketStatusUpdated status) {
    handleMarketStatus(status);
  } else if (event instanceof ProviderEvent.LifecycleUpdated lifecycle) {
    handleLifecycle(lifecycle);
  }
}
```

처음엔 `switch` 패턴 매칭으로 짜려다 **컴파일 에러** — switch pattern matching은
Java 21+, ADR-0015는 JDK 17 LTS. `instanceof` 패턴 변수 체인으로 내렸다. sealed
타입이라 default 없이도 "이 셋이 전부"지만, 컴파일러가 exhaustiveness를 100%
강제하진 못한다(17에선). 새 `ProviderEvent` 타입이 생기면 이 파일이 가장 먼저
손볼 곳이 되도록 리뷰 시점 안전망으로 둔다.

> **C/C++ 메모.** Java 17 `instanceof` 패턴 변수(`x instanceof Foo f`)는 캐스팅을
> 한 번에 한다 — C++17 `if (auto* f = dynamic_cast<Foo*>(x))`와 거의 동형.
> Java 21의 switch 패턴은 더 강력하지만 우리 LTS엔 없다.

### match-result

위 `handleLifecycle`의 끝 분기다. `LifecycleUpdated(FINISHED)`면
`provider.getMatchResult`를 불러 present일 때만 publish. real provider는 V1에서
empty를 반환하므로 if-present는 의도된 것(TODO 아님).

### single-instance 가정 (클래스에 명시)

V1은 orchestrator 프로세스 1개. 둘이면 모든 토픽에 double-publish. leader
election이나 event-id 샤딩은 Phase 5(k8s) 범위라 여기서 비용을 안 냈다. (클래스
Javadoc에 명시해 다음 단계가 발견하게 했다.)

## 결과

7개 테스트(mock 의존성). refresh가 이벤트당 1회만 구독, OddsUpdated →
publish+cache, MarketStatusUpdated → publish+cache, LifecycleUpdated →
publish+cache(status 갱신), FINISHED → getMatchResult+publish, **non-FINISHED는
getMatchResult 안 부름**(우발적 폴링 방지 negative test), 스트림 구독이 emit된
이벤트를 dispatch까지 전달(Sinks.Many → dispatch end-to-end).

## 요약

- provider→cache+publisher 조립, profile은 이 클래스에 갇힘.
- cache는 무조건 갱신(canonical), publish만 threshold 적용.
- discovery: PostConstruct + 30s refresh, 이벤트당 1구독.
- Java 17 → instanceof 체인(switch 패턴은 21+).
- single-instance 가정 명시.

## 기억·설명 Level

- **L1**: refresh 주기, 생명주기 훅.
- **L2**: 이벤트당 1구독(`computeIfAbsent`), lifecycle status 갱신 방식.
- **L3**: **cache는 무조건/​publish만 threshold** — 이 분리가 왜 정합성에
  중요한지(가장 자주 받는 질문). **Java 17 switch 패턴 부재 → instanceof**,
  **single-instance 가정과 그 한계**. 면접에서 "어떻게 조립했나"·"왜 cache는
  필터를 안 거치나" 단골.
