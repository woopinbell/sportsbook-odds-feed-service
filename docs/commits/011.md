# 011 — build(deps): Lettuce Redis 풀을 위한 commons-pool2 추가

## 개요

dev 단계가 아니라 **Phase 5 전체 스택(compose)**에서 드러난 부팅 크래시 수정. Spring Data Redis가 pooled Lettuce 클라이언트를 쓰는데(`spring.data.redis.lettuce.pool.enabled=true`), 그 풀이 요구하는 `org.apache.commons:commons-pool2`가 POM에 없어 `GenericObjectPoolConfig`에서 `ClassNotFoundException`이 났다. 단위 테스트는 unpooled 클라이언트로 캐시를 돌려 이 누락이 드러나지 않았고, 실제 풀을 켠 compose 런에서만 터졌다.

## 작업 순서

1. 누락 의존을 진단(부팅 로그의 `commons-pool2`/`GenericObjectPoolConfig` 미해결).
2. `pom.xml`에 `commons-pool2` 추가(버전은 Spring Boot BOM 관리).

## 작업 내역

spring-boot-starter-data-redis 바로 뒤에 의존을 추가하고, 왜 필요한지를 주석으로 박았다.

```xml
<!-- commons-pool2: Lettuce 커넥션 풀링(application.yml의 lettuce.pool.enabled=true)이
     요구. 통합 런타임에 없으면 GenericObjectPoolConfig에서 ClassNotFoundException.
     단위 테스트는 풀을 안 켜서 full-stack 런에서만 드러났다. -->
<dependency>
  <groupId>org.apache.commons</groupId>
  <artifactId>commons-pool2</artifactId>
</dependency>
```

## 결과

odds-feed가 compose에서 정상 부팅하고 Lettuce 풀이 commons-pool2로 커넥션을 관리한다.

## 요약

- pooled Lettuce는 commons-pool2를 런타임에 요구하는데 POM에 없어 부팅 크래시.
- 단위 테스트는 풀을 안 켜 누락이 안 드러나고 full-stack compose에서만 터짐.
- 의존 1줄 추가(버전 BOM 관리), 로직 무변경.

## 다음 작업

- 없음. cache·publisher 로직 무변경.

## 핵심 확인

- 진입점: `pom.xml`의 commons-pool2 의존, `application.yml`의 `spring.data.redis.lettuce.pool.enabled=true`.
- 단위 테스트가 못 잡은 이유: 풀을 안 켠 클라이언트로 캐시 테스트.

## 기억·설명 Level

- **L1**: pooled Lettuce엔 commons-pool2 의존이 필요하다.
- **L2**: 왜 단위 테스트에서 안 드러나고 compose에서만 났나(풀 enabled 차이).
- **L3**: "런타임에만 필요한 transitive 의존이 빠지면 단위 테스트로 안 잡히고 통합 환경에서 부팅 크래시로만 드러난다"는 통합 단계 결함 패턴.
