# 012 — fix(orchestrator): 마켓 상태를 lifecycle 양방향으로 캐시

## 개요

dev 단계가 아니라 **Phase 5 전체 스택 e2e**에서 드러난 결함 두 짝이다. betting-service는 슬립을 받기 전 Redis의 `market:{e}:{m} == OPEN`을 확인한다. 그런데 odds-feed는 마켓 상태를 **명시적 MarketStatusUpdated(suspend/close) 이벤트에서만** 캐시에 썼다 — 정상적으로 열려 있는 마켓은 이 키가 아예 없어서 **모든 베팅이 "market not open"으로 거절**됐다(버그 #3). 거울상도 있었다: 끝난/연기된/취소된 이벤트가 키를 영영 OPEN으로 남겨 betting이 죽은 이벤트에도 슬립을 받았다. 둘 다 단위 테스트는 키를 손으로 seed해서 못 잡고 full-stack e2e에서만 드러났다.

## 작업 순서

1. odds tick마다 OPEN을 캐시(odds는 열린 마켓에만 흐른다).
2. 이벤트별로 본 마켓을 추적(`marketsByEvent`).
3. terminal lifecycle(FINISHED / CANCELLED / POSTPONED)에서 그 이벤트의 마켓을 CLOSE.

## 작업 내역

### OPEN을 odds tick에 쓴다

provider는 마켓이 OPEN일 때만 odds를 흘리므로, odds 업데이트는 곧 "이 마켓은 열려 있다"는 신호다. `handleOdds`에서 odds를 캐시할 때 `storeMarketStatus(OPEN)`도 같이 쓰고, 본 (event, market)을 `marketsByEvent`에 추적한다(CLOSE 대상을 알기 위해). market-status 이벤트 경로에서도 같은 추적을 한다:

```java
// 신규 필드: 이벤트별 본 마켓 — terminal lifecycle이 닫을 대상.
private final Map<EventId, Set<MarketId>> marketsByEvent = new ConcurrentHashMap<>();

private void handleOdds(ProviderEvent.OddsUpdated event) {
  publisher.publishOddsChanged(event.eventId(), event.marketId(), event.selectionId(),
      event.previousOdds(), event.newOdds(), event.occurredAt());
  cache.storeOdds(event.eventId(), event.marketId(), event.selectionId(), event.newOdds());
  cache.storeMarketStatus(event.eventId(), event.marketId(), MarketStatus.OPEN);  // 추가: odds=열림 신호
  trackMarket(event.eventId(), event.marketId());                                 // 추가
}

private void handleMarketStatus(ProviderEvent.MarketStatusUpdated event) {
  publisher.publishMarketStatusChanged(event.eventId(), event.marketId(),
      event.previousStatus(), event.newStatus(), event.reason(), event.occurredAt());
  cache.storeMarketStatus(event.eventId(), event.marketId(), event.newStatus());
  trackMarket(event.eventId(), event.marketId());                                 // 추가
}

private void trackMarket(EventId eventId, MarketId marketId) {
  marketsByEvent.computeIfAbsent(eventId, k -> ConcurrentHashMap.newKeySet()).add(marketId);
}
```

### terminal lifecycle에서 CLOSE (거울상)

OPEN만 쓰고 안 지우면 죽은 이벤트가 OPEN으로 남는다. `handleLifecycle`에서 status가 terminal이면 그 이벤트의 추적된 마켓을 CLOSED로 뒤집고 `MarketStatusChanged`를 publish한다. 직전 상태를 읽어 정확한 전이를 publish하고, 이미 CLOSED면 skip(재전달 멱등):

```java
private void handleLifecycle(ProviderEvent.LifecycleUpdated event) {
  // ... publishEventLifecycle + cache/catalog status 갱신 (007/008과 동일) ...

  if (isTerminal(event.status())) {                       // 추가: 죽은 이벤트 마켓 닫기
    closeMarkets(event.eventId(), event.status(), event.occurredAt());
  }
  if (event.status() == EventLifecycleStatus.FINISHED) {  // 기존: match result publish
    provider.getMatchResult(event.eventId()).ifPresent(o -> publisher.publishMatchResult(...));
  }
}

private static boolean isTerminal(EventLifecycleStatus status) {
  return status == EventLifecycleStatus.FINISHED
      || status == EventLifecycleStatus.CANCELLED
      || status == EventLifecycleStatus.POSTPONED;
}

private void closeMarkets(EventId eventId, EventLifecycleStatus status, Instant occurredAt) {
  for (MarketId marketId : marketsByEvent.getOrDefault(eventId, Set.of())) {
    MarketStatus previous = cache.getMarketStatus(eventId, marketId).orElse(MarketStatus.OPEN);
    if (previous == MarketStatus.CLOSED) continue;        // 멱등
    publisher.publishMarketStatusChanged(
        eventId, marketId, previous, MarketStatus.CLOSED, "EVENT_" + status, occurredAt);
    cache.storeMarketStatus(eventId, marketId, MarketStatus.CLOSED);
  }
}
```

`marketsByEvent`(in-memory `ConcurrentHashMap`, 값은 `newKeySet()`)가 (event → markets)를 추적해 CLOSE 대상을 안다. reason은 `"EVENT_" + status`(예: `EVENT_FINISHED`)로 betting/settlement이 사유를 읽게 했다.

## 결과

- betting이 정상 마켓에 슬립을 받는다(OPEN이 캐시됨). 라이브 검증: 종료 마켓에 베팅 → 422 EVENT_CLOSED.
- 테스트: `terminalLifecycleClosesTrackedMarkets`(OPEN→CLOSED publish + cache), `terminalLifecycleWithNoKnownMarketsClosesNothing`(추적 0이면 무동작).

## 요약

- 정상 OPEN 마켓이 캐시에 없어 전건 거절(버그 #3) → odds tick마다 OPEN 캐시.
- 거울상: 죽은 이벤트가 OPEN으로 잔존 → terminal lifecycle에서 CLOSE + publish.
- `marketsByEvent`로 이벤트별 마켓 추적, CLOSE는 멱등.
- 단위 테스트는 키를 손으로 seed해 못 잡고 full-stack e2e에서만 드러남.

## 기억·설명 Level

- **L1**: odds tick에 OPEN 쓰기, terminal lifecycle에 CLOSE.
- **L2**: `marketsByEvent` 추적, `MarketStatusChanged` publish, 멱등 CLOSE, 직전 상태 읽어 전이 표시.
- **L3**: 마켓 상태 캐시가 한 방향(suspend/close 이벤트)만이라 정상 OPEN이 누락된 구조와 그 거울상(죽은 이벤트 OPEN 잔존). betting의 OPEN 게이트와의 cross-service 계약. 단위 테스트가 키를 seed해 못 잡고 e2e에서만 드러난 점.
