# 005 — feat(cache): Redis write-through cache for current odds and metadata

## 개요

하류 서비스가 Kafka 업데이트 **사이사이에** 읽는 거울(mirror). betting-service는
슬립 검증 때 현재 odds를, gateway는 WebSocket hello 프레임용 이벤트 메타데이터를,
admin-api는 마켓 상태를 본다. 셋 다 HTTP로 우리에게 묻는 대신 **Redis를 read**해서,
업데이트 사이 우리 가용성에 안 묶이게 한다(repo CLAUDE.md write-through 결정).

## 작업 순서

1. `CacheProperties` — `oddsfeed.cache.ttl` 바인딩.
2. `CacheKeys` — key 문자열 생성 중앙화.
3. `RedisOddsCache` — store/get 3쌍.
4. `ApplicationConfig`에 `CacheProperties` 등록.
5. Testcontainers Redis 통합 테스트.

## 작업 내역

### key 구조 (CLAUDE.md 결정, CacheKeys에 박제)

```
odds:{eventId}:{marketId}:{selectionId}  → "1.8500"  (Odds.SCALE decimal 문자열)
event:{eventId}                          → EventSummary JSON
market:{eventId}:{marketId}              → MarketStatus enum 이름
```

key 생성을 한 곳에 모아 rename이 단일 파일 변경이 되게 했다:

```java
public final class CacheKeys {
  private CacheKeys() {}

  public static String odds(EventId eventId, MarketId marketId, SelectionId selectionId) {
    return "odds:" + eventId.value() + ":" + marketId.value() + ":" + selectionId.value();
  }
  public static String event(EventId eventId) {
    return "event:" + eventId.value();
  }
  public static String market(EventId eventId, MarketId marketId) {
    return "market:" + eventId.value() + ":" + marketId.value();
  }
}
```

세 key 모두 단일 24h TTL(V1). 메타데이터 cache miss가 실제 비용이 될 때만
per-key TTL로 쪼갠다:

```java
@ConfigurationProperties(prefix = "oddsfeed.cache")
public record CacheProperties(Duration ttl) {}
```

### 저장 포맷 결정

세 종류 데이터를 각각 다른 포맷으로 store/get한다 — 직렬화 비용과 hot-path
빈도를 맞췄다:

```java
// odds — 평문 decimal 문자열 (Odds JSON 아님). betting-service hot path.
public void storeOdds(EventId e, MarketId m, SelectionId s, Odds odds) {
  redis.opsForValue().set(CacheKeys.odds(e, m, s), odds.decimal().toPlainString(), ttl);
}
public Optional<Odds> getOdds(EventId e, MarketId m, SelectionId s) {
  String v = redis.opsForValue().get(CacheKeys.odds(e, m, s));
  return v == null ? Optional.empty() : Optional.of(Odds.ofDecimal(v));
}

// event — JSON (중첩 구조: sport, status, kickoff Instant).
public void storeEvent(EventSummary summary) {
  try {
    redis.opsForValue().set(
        CacheKeys.event(summary.eventId()), objectMapper.writeValueAsString(summary), ttl);
  } catch (JsonProcessingException e) {
    throw new IllegalStateException("Failed to serialize EventSummary", e);
  }
}
public Optional<EventSummary> getEvent(EventId eventId) {
  String json = redis.opsForValue().get(CacheKeys.event(eventId));
  if (json == null) return Optional.empty();
  try {
    return Optional.of(objectMapper.readValue(json, EventSummary.class));
  } catch (JsonProcessingException e) {
    throw new IllegalStateException("Failed to deserialize EventSummary", e);
  }
}

// market status — enum 이름 (JSON·ordinal 아님).
public void storeMarketStatus(EventId e, MarketId m, MarketStatus status) {
  redis.opsForValue().set(CacheKeys.market(e, m), status.name(), ttl);
}
public Optional<MarketStatus> getMarketStatus(EventId e, MarketId m) {
  String v = redis.opsForValue().get(CacheKeys.market(e, m));
  return v == null ? Optional.empty() : Optional.of(MarketStatus.valueOf(v));
}
```

- **odds는 평문 decimal 문자열**(Odds JSON 아님). betting-service는 가격만
  필요한데 Jackson 직렬화는 hot path에서 바이트·파싱을 배로 만든다. decimal은
  `Odds.SCALE`를 보존하므로 round-trip이 정확("1.85" → "1.8500").
- **`EventSummary`만 JSON** — 중첩 구조(sport, status, kickoff Instant)가 있어서.
  테스트의 `ObjectMapper`는 `JavaTimeModule`을 명시 등록해 Spring 컨텍스트 밖에서도
  `Instant`가 round-trip되게 했다.
- **`MarketStatus`는 enum 이름** (`name()`/`valueOf()`). Avro 스키마 진화에도
  살아남고 `redis-cli`에서 사람이 읽힌다.

> **C/C++ 메모.** enum을 ordinal(정수)로 저장하면 enum 순서를 바꾸는 순간 기존
> 데이터가 깨진다. **이름(name)으로 저장**하면 순서 변경에 안전하다 — C/C++에서
> enum 값을 직렬화할 때 정수 대신 문자열을 쓰는 것과 같은 이유.

### write-through 정직함

`@Component`로 등록되고 생성자에서 `StringRedisTemplate` + `ObjectMapper` +
TTL을 받는다. odds 저장은 단 한 줄(`set`)이라 atomic 보장이 없다:

```java
@Component
public class RedisOddsCache {
  private final StringRedisTemplate redis;
  private final ObjectMapper objectMapper;
  private final Duration ttl;

  public RedisOddsCache(StringRedisTemplate redis, ObjectMapper objectMapper, CacheProperties props) {
    this.redis = redis;
    this.objectMapper = objectMapper;
    this.ttl = props.ttl();
  }
  // ... 위의 store/get 3쌍
}
```

orchestrator(007)가 publisher → cache 순으로 쓴다. publish 성공 후 cache 실패면
Kafka가 authoritative이고 다음 변경이 둘을 덮어쓴다. Kafka+Redis 진짜 원자성은
transactional outbox가 필요한데, 그 패턴은 **돈을 다루는** wallet/betting의
몫(ADR-0006)이다. odds-feed 이벤트는 멱등적(누락된 캐시 갱신은 그 selection의 바로
다음 변경이 정정)이라 outbox 세금을 여기선 명시적으로 안 냈다 — V1
비용-정합성 trade-off 결정.

## 결과

5개 테스트(누적 38). Testcontainers Redis(임베디드 아닌 실제 wire 프로토콜,
sportsbook/CLAUDE.md Level 1): odds round-trip(정확한 decimal 문자열), missing-key
empty, EventSummary JSON round-trip, MarketStatus round-trip, 그리고 **CacheKeys
포맷 잠금 테스트**(key 레이아웃 rename은 의도적으로만 가능하게).

## 요약

- 3가지 key, 단일 24h TTL.
- odds=평문 decimal(hot path), EventSummary=JSON, MarketStatus=enum 이름.
- write-through(non-atomic) — 멱등성으로 정당화, outbox는 안 씀.
- Testcontainers Redis로 실제 프로토콜 검증.

## 기억·설명 Level

- **L1**: 3가지 key 구조, TTL 24h.
- **L2**: 저장 포맷별 선택(decimal/JSON/enum 이름) 이유, key 중앙화.
- **L3**: **write-through가 non-atomic인데 왜 괜찮은가**(odds 이벤트의 멱등성
  vs 돈을 다루는 서비스의 outbox 필요성) — ADR-0006 경계 설명. 면접에서
  "캐시 일관성"·"왜 여기엔 outbox를 안 썼나" 단골.
