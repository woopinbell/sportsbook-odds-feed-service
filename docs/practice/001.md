# 001 — feat(provider): OddsProvider abstraction and DTO types

## 개요

ADR-0010이 요구하는 **seam(이음매)** 을 만든다. `OddsProvider` 인터페이스 하나
뒤에 mock(기본)과 real(옵션) 두 adapter가 선다. 이 커밋은 **타입만** 추가하고
구현은 다음 두 커밋에서 채운다. 외부 의존을 인터페이스로 격리해 테스트
친화적으로 만드는, hexagonal/clean architecture의 외부 어댑터 패턴 그대로다.

## 작업 순서

1. `Sport` enum (FOOTBALL, BASKETBALL) — 로컬 유지.
2. `EventSummary` 레코드 — 이벤트 스냅샷.
3. `ProviderEvent` sealed interface — 변경 이벤트 3종.
4. `MatchOutcome` 레코드 — 경기 결과.
5. `OddsProvider` 인터페이스 — 3메서드.
6. 단위 테스트 — sealed-permits 계약, null guard, 방어적 복사.

## 작업 내역

### OddsProvider 인터페이스 — ADR-0010 스케치에서 의도적으로 바꾼 것

계약이라 시그니처는 노출. **왜 이 반환 타입인지**를 설명할 수 있어야 한다:

```java
public interface OddsProvider {
  List<EventSummary> listEvents(Sport sport);
  Flux<ProviderEvent> streamEvents(EventId eventId);
  Optional<MatchOutcome> getMatchResult(EventId eventId);
}
```

**책임:** `listEvents`가 `Stream`이 아니라 `List`인 이유(스냅샷을 여러 번 훑음),
`streamEvents`가 셋이 아니라 단일 `Flux<ProviderEvent>`인 이유(발생 순서 보존)를
근거로 댈 수 있어야 한다.

### ProviderEvent — sealed interface (Java 17)

이 커밋의 핵심 재유도 단위. `permits` 봉인은 노출하되, 각 변형이 **무슨 필드를
나르고 무엇을 검증하는지**는 비운다:

```java
public sealed interface ProviderEvent
    permits ProviderEvent.OddsUpdated,
        ProviderEvent.MarketStatusUpdated,
        ProviderEvent.LifecycleUpdated {

  EventId eventId();
  Instant occurredAt();

  /** 한 selection 가격 이동. publish 시 1% 임계값 적용. */
  record OddsUpdated(/* TODO: 가격 이동이 나를 필드 */) implements ProviderEvent {
    public OddsUpdated {
      // TODO: 필드 non-null 검증
    }
  }

  /** 마켓 open/suspend/close. */
  record MarketStatusUpdated(/* TODO: 마켓 상태 전이 필드, reason 포함 */) implements ProviderEvent {
    public MarketStatusUpdated {
      // TODO: non-null 검증 — reason 하나만 예외(왜?)
    }
  }

  /** 이벤트 lifecycle 전이(kickoff/종료/취소). */
  record LifecycleUpdated(/* TODO: lifecycle 전이 필드 */) implements ProviderEvent {
    public LifecycleUpdated {
      // TODO: 필드 non-null 검증
    }
  }
}
```

**책임:** 변경 3종을 한 sealed union으로 컴파일 타임 봉인(publisher가 instanceof로
dispatch). 각 변형의 필드 구성과 null 계약을 정한다 — `reason`만 검증에서 빼는
이유(open 전이엔 사유가 없을 수 있음)를 설명할 수 있어야.

### DTO 레코드 + shared-protocol 타입 재사용

record 이름·필드는 다른 커밋이 쓰는 공개 계약이라 노출. 검증·방어 복사·로컬
enum 결정은 비운다:

```java
public record EventSummary(
    EventId eventId, Sport sport, String competition,
    String homeTeam, String awayTeam,
    Instant scheduledStartAt, EventLifecycleStatus status) {
  public EventSummary {
    // TODO: 모든 필드 non-null
  }
}

public record MatchOutcome(
    EventId eventId, String score, MatchFinalStatus finalStatus,
    Map<String, String> detail, Instant settledAt) {
  public MatchOutcome {
    // TODO: non-null 검증 + detail을 방어 복사(호출자가 나중에 못 바꾸게)
  }
}

public enum Sport {
  // TODO: V1 종목 (이 enum을 shared-protocol이 아니라 로컬에 두는 이유는?)
}
```

**책임:** record는 불변 데이터 캐리어. `MatchOutcome.detail`은 방어 복사로
캡슐화(schema-less Map). `Sport`는 로컬 enum — 다른 서비스는 `eventId`로만
식별하므로 shared-protocol을 오염시키지 않는다. 재사용 타입(`Odds`/`EventId`/
도메인 enum)이 주는 이점(scale 정규화 equality, 컴파일 타임 타입 구분)을 안다.
