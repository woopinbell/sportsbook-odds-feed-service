# 008 — feat(api): REST read endpoints and trader admin endpoints

## 개요

HTTP 클라이언트를 위한 두 표면적:

```
GET  /api/v1/events                              kickoff 순 cursor pagination
GET  /api/v1/events/{id}                         단건
GET  /api/v1/odds/{eventId}/{marketId}/{selId}   현재 odds 조회

POST /internal/v1/events/{eventId}/markets/{marketId}/suspend
POST /internal/v1/events/{eventId}/markets/{marketId}/close
POST /internal/v1/events/{eventId}/markets/{marketId}/reopen
```

`/api/v1`은 고객용(gateway 라우팅), `/internal/v1`은 admin-api용. path로 가르면
미래의 API gateway가 endpoint별 애너테이션 없이 **경로 단위로 보안 정책**을
다르게 줄 수 있다.

## 작업 순서

1. `EventCatalog` — 인메모리 이벤트 인덱스.
2. `CursorPage<T>` + `EventReadController` — cursor pagination.
3. `OddsReadController` — Redis read.
4. `MarketAdminController` — suspend/close/reopen.
5. `FeedOrchestrator`에 catalog 4번째 인자로 연결.
6. `@WebMvcTest` 컨트롤러 테스트.
