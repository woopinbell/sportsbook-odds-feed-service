# 007 — feat(orchestrator): wire provider, cache, and publisher together

## 개요

`FeedOrchestrator`는 이 서비스의 세 부품(provider·cache·publisher)이 다 존재함을
아는 **유일한** 조립 지점이다. 활성 `OddsProvider`(profile에 따라 mock 또는
real)에서 `EventSummary`를 끌어와, 이벤트별 `ProviderEvent` 스트림을 구독하고,
각 이벤트를 알맞은 cache + publish 경로로 보낸다. **profile 선택이 이 클래스를
넘어 전파되지 않는다** — cache도 publisher도 자기가 mock을 섬기는지 real을
섬기는지 모른다. adapter 패턴(001)의 배당금이 여기서 실현된다.

## 작업 순서

1. 4-인자 생성자(provider, cache, publisher, + 008에서 catalog).
2. discovery 루프 — `@PostConstruct` + `@Scheduled refresh()`.
3. 이벤트별 `streamEvents` 구독(중복 방지).
4. `dispatch` — sealed `ProviderEvent` 분기 → cache+publish.
5. 7개 mock 기반 테스트.

**책임:** cache 갱신은 **모든** 이벤트에 무조건(canonical — betting이 가장 신선한

lifecycle은 status만 교체(kickoff·팀명 보존, provider 재조회 불필요).

### match-result

**책임:** `handleLifecycle`의 끝 분기 — `LifecycleUpdated(FINISHED)`면
`getMatchResult`를 불러 present일 때만 publish. real provider는 V1에서 empty를
반환하므로 if-present는 의도된 것(TODO 아님).
