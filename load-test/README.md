# 배당 조회 부하 검증

발행 처리량과 HTTP 조회 성능을 서로 다른 검사로 다룹니다.

- `KafkaPublishThroughputTest`는 내장 브로커에서 발행기 자체의 처리량을 확인합니다.
- `run-http-gate.sh`는 실제 Redis와 Kafka를 새로 실행하고 경기 목록과 한 배당 조회
  API의 릴리스 조건을 확인합니다.

## 준비

Docker, Java, k6, `curl`, `jq`, `nc`, `redis-cli`, `shasum`이 필요합니다. 공통
계약을 격리된 Maven 저장소에 먼저 설치합니다.

```sh
(cd ../sportsbook-shared-protocol-fix && ./mvnw \
  -Dmaven.repo.local=/tmp/odds-release-m2 clean install)

MAVEN_REPO_LOCAL=/tmp/odds-release-m2 \
  ./load-test/run-http-gate.sh
```

실행기는 수정된 작업 트리를 거부하고 시작·종료 커밋, 트리, JAR, 실행 스크립트,
k6 시나리오, Compose 파일을 결과에 기록합니다. 결과는 기본적으로
`load-test/results/gate/<run-id>/`에 저장됩니다. 저장소 밖에 두려면 쓰기 가능한
절대 경로를 `RESULT_ROOT`로 지정합니다.

## 측정 방식

경기 목록과 한 배당 조회를 서로 독립된 인프라 상태에서 검사합니다. 각 경로는 다음
순서로 실행합니다.

1. 초당 1,000회로 60초 동안 예열합니다.
2. 초당 1,000회, 60초 측정을 다섯 번 반복합니다.
3. 각 구간의 앞뒤에서 응답 계약과 기준 데이터를 다시 확인합니다.

각 측정은 p99 50ms 미만, 오류율 0.1% 미만, 검사 성공률 99.9% 초과, 누락 반복 0건을
모두 만족해야 합니다. 두 경로를 동시에 호출하는 구간은 병목 관찰용이며 합격 조건에는
포함하지 않습니다.

로컬 OTLP 수집기가 없을 때 내보내기 재시도가 응답 시간을 왜곡하지 않도록 부하 실행은
`OTEL_SAMPLING_PROBABILITY=0`을 사용합니다. 관측 비용은 실제 수집기를 둔 별도
환경에서 확인해야 합니다. `JFR_OUTPUT_DIR`에 쓰기 가능한 절대 경로를 지정하면 각
서비스 구간의 제한된 JFR 기록도 남길 수 있습니다.

## 확인된 결과

2026-07-13 실행에서 두 경로가 모두 조건을 통과했습니다.

- 경기 목록 p99: 9.503, 1.743, 2.440070, 3.335, 1.535ms
- 한 배당 조회 p99: 5.961, 8.969, 8.874070, 17.883, 17.289ms
- 모든 측정의 HTTP 오류 0건, 검사 성공률 100%, 누락 반복 0건

이 결과는 부하 전용으로 OpenTelemetry 표본 수집을 끈 로컬 환경에만 적용됩니다.
운영 수용량을 판단하려면 고정된 자원, 별도 부하 생성기, 실제 관측 수집기를 사용해
다시 측정해야 합니다.

## 발행기 기준

```sh
./mvnw test -Dtest=KafkaPublishThroughputTest
```

단일 JVM과 내장 브로커에서 초당 100,000건을 넘는지 확인합니다. 네트워크를 거치는
Kafka 클러스터의 수용량과 구분해 해석해야 합니다.
